Remove LICM Phase 4 sub-expression extraction to fix SIGSEGV

Phase 4 of the Loop-Invariant Code Motion optimization in Repeated.java
extracted invariant sub-expressions from variant expressions and replaced
them with references to new hoisted declarations. This produced invalid
generated C code that caused a JVM SIGSEGV crash in native JNI execution
during AggregatedComputationTests (sum reduction over PackedCollection).

The crash was confirmed as a regression: the test passes on master and
crashes on this branch with Phase 4 enabled. Disabling Phase 4 while
keeping Phases 1-3 (basic declaration hoisting) resolves the crash.

Phase 4 corresponds to optimization #6 in the AUDIO_SCENE_LOOP plan
("Extract + hoist genome sub-expressions from envelope lines"), which
was listed as "Not started — depends on CSE + LICM interaction",
indicating it was added prematurely.

Changes:
- Removed Phase 4 invocation from hoistLoopInvariantStatements()
- Removed all Phase 4 helper methods: extractInvariantSubExpressions,
  extractFromScope, extractSubExpressionsFromAssignment,
  replaceInvariantSubExpressions, createAssignment,
  findMaximalInvariantSubExpressions, isSubstantialForExtraction
- Cleaned up unused imports (Constant, HashMap, Map)
