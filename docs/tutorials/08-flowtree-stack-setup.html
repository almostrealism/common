Tutorial: Setting Up the FlowTree Coding Agent Stack

Home Tutorials FlowTree Module

Introduction

This tutorial walks through deploying the complete FlowTree coding agent stack from scratch: the controller, memory server, API bridge, and a pool of Claude Code agents. By the end you will have:

  • A flowtree-controller accepting agent connections and dispatching ClaudeCodeJob instances
  • An ar-memory semantic store persisting agent memories across sessions
  • An ar-manager MCP server reachable from the public internet via Tailscale Funnel, acting as the primary entrypoint for submitting jobs and querying status
  • A pool of agent containers that connect to the controller, run Claude Code headlessly, and push git changes

Prerequisites

  • Docker Desktop (or Docker Engine + Compose plugin) installed
  • Java 17 and Maven installed (for the build step)
  • Tailscale installed and authenticated on the controller host
  • A Claude Code OAuth token — run claude setup-token
  • A GitHub personal access token with repo and workflow scopes (for Tier 2 pipeline operations)
  • The common repository cloned locally

Architecture Overview

External triggers (Slack, CI, API, Claude mobile) │ │ HTTPS via Tailscale Funnel ▼ ┌────────────────────────────────────────────────────────┐ │ Controller Stack (flowtree/controller/docker-compose) │ │ │ │ ┌──────────────────────┐ ┌──────────────────────┐ │ │ │ flowtree-controller │ │ ar-manager │ │ │ │ │◄──│ (MCP bridge) │ │ │ │ :7766 agent port │ │ :8010 ◄── Funnel │ │ │ │ :7780 internal API │ │ │ │ │ └──────────┬───────────┘ └──────────────────────┘ │ │ │ │ │ │ │ ┌──────────▼─────────┐ │ │ │ │ ar-memory │ │ │ │ │ :8020 │ │ │ │ └─────────────────────┘ │ └─────────────┼──────────────────────────────────────────┘ │ outbound TCP :7766 ▼ ┌────────────────────────────────────────────────────────┐ │ Agent Pool (flowtree/agent/docker-compose) │ │ │ │ ┌───────────┐ ┌───────────┐ ┌───────────┐ │ │ │ agent-1 │ │ agent-2 │ │ agent-N │ │ │ │ Claude │ │ Claude │ │ Claude │ │ │ │ Code CLI │ │ Code CLI │ │ Code CLI │ │ │ └───────────┘ └───────────┘ └───────────┘ │ └────────────────────────────────────────────────────────┘

flowtree-controller

:7766 (agents) · :7780 (internal API)

Java server. Accepts outbound connections from agents, dispatches ClaudeCodeJob instances, integrates with Slack. Internal only — do not call port 7780 directly.

ar-manager

:8010 (public via Tailscale Funnel)

Primary entrypoint. Python MCP server. Exposes FlowTree to Claude mobile, CI pipelines, and Claude Code agents via authenticated HTTP. Handles GitHub, memory, and workstream management.

ar-memory

:8020 (internal)

Semantic vector store. Persists agent memories and decisions across sessions so agents recall prior work and avoid repeating mistakes.

Agent containers

no inbound ports required

Each agent connects outbound to the controller. Runs Claude Code headlessly, commits changes, and pushes to the target git branch. Scales horizontally.

Why ar-manager and not the controller API directly? The controller's port 7780 is a raw internal API with no authentication, rate limiting, or GitHub integration. ar-manager adds all of those and also serves as a Claude Code MCP server, making it the correct entrypoint for every external caller.

1 Understand the directory layout

All FlowTree infrastructure lives under flowtree/ in the repository:

flowtree/ controller/ Dockerfile ← flowtree-controller image docker-compose.yml ← controller stack (controller + ar-memory + ar-manager) agent/ Dockerfile ← agent image docker-compose.yml ← agent pool .env.example ← copy to .env and fill in start.sh conf/ agent.properties ← agent runtime configuration rebuild.sh ← one-command build + deploy tools/mcp/ manager/ ← ar-manager source (Python) setup.sh ← Tailscale Funnel setup helper generate-token.sh ← creates a bearer token memory/ ← ar-memory source (Python)

The rebuild.sh script at the repository root handles the Maven build and coordinates both stacks. Always run it from the repository root.

2 Enable Tailscale Funnel

Tailscale Funnel gives ar-manager a stable public HTTPS URL with automatic TLS — no certificate management required. This step must be done before starting the stack so the container can receive its public URL as an environment variable.

2a. Install and authenticate Tailscale

# macOS
brew install tailscale
open -a Tailscale   # follow the login prompt

# Linux
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
tailscale status   # confirm you see your machine listed

2b. Enable HTTPS and Funnel in the admin console

  1. Log in at login.tailscale.com/admin
  2. Go to DNS → enable HTTPS Certificates
  3. Go to Access Controls → enable Funnel
Funnel requires a Tailscale Personal or Team plan. The free plan supports Funnel for evaluation. Once enabled in the admin console, the CLI command tailscale funnel becomes available.

2c. Note your Tailscale DNS name

tailscale status --self --json | python3 -c \
  "import json,sys; print(json.load(sys.stdin)['Self']['DNSName'].rstrip('.'))"
# e.g. my-studio.taild1234.ts.net

This is the public hostname where ar-manager will be reachable. Keep it handy — you will use it when configuring Claude mobile.

3 Create host directories and configuration files

The controller stack expects configuration and secrets on the host machine at known paths mounted as Docker volumes.

sudo mkdir -p /Users/Shared/flowtree/{controller,manager,memory-data,secrets}
sudo chown -R "$(whoami)" /Users/Shared/flowtree

3a. Workstreams configuration

Copy the example file and edit it:

cp flowtree/src/main/resources/workstreams-example.yaml \
   /Users/Shared/flowtree/controller/workstreams.yaml

At minimum, configure one workstream. Key fields:

FieldDescription
idShort identifier, e.g. ws-myproject
repoUrlSSH clone URL of the target repository
defaultBranchBranch agents check out by default
githubOrgGitHub org name (for PR creation)
allowedToolsComma-separated Claude tool allowlist

3b. GitHub token (optional, for PR creation)

If you want agents to create pull requests, set AR_MANAGER_GITHUB_TOKEN in the environment before running rebuild.sh, or add it to a .env file the compose file can pick up:

export AR_MANAGER_GITHUB_TOKEN=ghp_your_token_here
The shared secret between ar-manager and the controller is generated automatically by rebuild.sh on the first run and written to /Users/Shared/flowtree/secrets/shared-secret. You do not need to create it manually.

4 Build and start the controller stack

Run from the repository root:

./flowtree/rebuild.sh

This script:

  1. Generates the ar-manager shared secret (first run only)
  2. Runs mvn package -pl flowtree -am -DskipTests
  3. Copies transitive dependencies to target/dependency/
  4. Builds and starts flowtree-controller, ar-memory, and ar-manager via Docker Compose

Verify all three services are running

docker compose -f flowtree/controller/docker-compose.yml ps

Expected output:

NAME                  STATUS          PORTS
flowtree-controller   running         0.0.0.0:7766->7766/tcp, 0.0.0.0:7780->7780/tcp
ar-memory             running         0.0.0.0:8020->8020/tcp
ar-manager            running         0.0.0.0:8010->8010/tcp
# Health checks
curl http://localhost:7780/health          # controller
curl http://localhost:8020/_health         # ar-memory
curl http://localhost:8010/_health         # ar-manager
Rebuilding a single service after a code change: 
./flowtree/rebuild.sh flowtree-controller
./flowtree/rebuild.sh ar-manager

5 Expose ar-manager via Tailscale Funnel

With the controller stack running, punch ar-manager through to the internet:

./tools/mcp/manager/setup.sh

This script:

  1. Generates a bearer token into /Users/Shared/flowtree/manager/manager-tokens.json (if none exists)
  2. Derives your Tailscale DNS name and sets AR_MANAGER_ISSUER_URL
  3. Rebuilds and restarts the ar-manager container
  4. Runs tailscale funnel --bg 8010

When it finishes:

==> Funnel active. Public MCP endpoint:

    https://my-studio.taild1234.ts.net/

    Configure this URL in Claude mobile as a remote MCP server.
    OAuth will prompt you to enter your bearer token.

Verify public access

tailscale funnel status
curl https://my-studio.taild1234.ts.net/_health

Add more bearer tokens

Each client (Claude mobile, a CI pipeline, a teammate) should have its own token with appropriate scopes:

# Full access token (read + write + pipeline + memory)
AR_MANAGER_TOKEN_FILE=/Users/Shared/flowtree/manager/manager-tokens.json \
  ./tools/mcp/manager/generate-token.sh "My label"

# Read-only token
AR_MANAGER_TOKEN_FILE=/Users/Shared/flowtree/manager/manager-tokens.json \
  ./tools/mcp/manager/generate-token.sh "Monitoring" --scopes read,memory

Token scopes:

ScopeGrants access to
readList workstreams, get stats, health check
writeSubmit tasks, register/update workstreams
pipelineTrigger GitHub workflows, commit plan files
memoryRecall, store, and browse agent memories
Security: The bearer token is the only credential protecting ar-manager. Treat it like a password. Tailscale Funnel handles TLS — never add a plain HTTP Funnel or expose port 8010 directly to the internet without Funnel.

6 Start the agent pool

Agents can run on the same machine as the controller or on a separate host. They connect outbound to the controller, so no inbound firewall rules are needed on the agent machine.

6a. Configure the agent environment

cp flowtree/agent/.env.example flowtree/agent/.env

Edit flowtree/agent/.env:

FLOWTREE_ROOT_HOST=my-studio.taild1234.ts.net   # controller hostname
FLOWTREE_ROOT_PORT=7766                         # default, change if needed
CLAUDE_CODE_OAUTH_TOKEN=<token from claude setup-token>
GIT_USER_NAME=FlowTree Agent
GIT_USER_EMAIL=agent@yourorg.com
AGENT_COUNT=2                                   # number of parallel agents
SSH_KEY_DIR=~/.ssh                              # SSH key for git push

6b. Start the pool

./flowtree/rebuild.sh --agents-only

Or, to build both stacks together in one command:

./flowtree/rebuild.sh --agents

6c. Verify agents are connected

docker compose -f flowtree/agent/docker-compose.yml ps
docker compose -f flowtree/agent/docker-compose.yml logs -f

Look for log lines like:

FlowTree Agent starting
  Controller: my-studio.taild1234.ts.net:7766
NodeGroup: Added relay node (index 1)
NodeGroup: startPersistentHost connecting to my-studio.taild1234.ts.net:7766
Scale up or down without rebuilding: 
AGENT_COUNT=4 docker compose -f flowtree/agent/docker-compose.yml up -d --scale agent=4

7 Submit your first job

With the stack running, submit a job through ar-manager:

Via HTTP (curl)

curl -X POST https://my-studio.taild1234.ts.net/api/workstreams/ws-myproject/submit \
  -H "Authorization: Bearer armt_your_token" \
  -H "Content-Type: application/json" \
  -d '{
    "prompt": "Add a missing null check to UserService.findById()",
    "targetBranch": "feature/null-checks"
  }'

Via Claude Code MCP tool

Configure ar-manager as a remote MCP server in your Claude Code session (add to .mcp.json or settings.local.json):

{
  "mcpServers": {
    "ar-manager": {
      "type": "http",
      "url": "https://my-studio.taild1234.ts.net/",
      "headers": {
        "Authorization": "Bearer armt_your_token"
      }
    }
  }
}

Then use the MCP tool in a Claude Code session:

workstream_submit_task(
    prompt="Add a missing null check to UserService.findById()",
    workstream_id="ws-myproject",
    target_branch="feature/null-checks"
)

Via Claude mobile

  1. Open Claude mobile → Settings → MCP Servers
  2. Add a new server with URL https://my-studio.taild1234.ts.net/
  3. Authenticate with your bearer token when prompted
  4. Use natural language: "Submit a task to ws-myproject to add null checks to UserService"

Monitor progress

# Check job status via API
curl -H "Authorization: Bearer armt_your_token" \
  "https://my-studio.taild1234.ts.net/api/workstreams/ws-myproject/status"

# Watch agent logs in real time
docker compose -f flowtree/agent/docker-compose.yml logs -f

8 Surviving reboots

Tailscale Funnel configuration persists across reboots automatically. Docker containers do not — they need to be restarted. The simplest approach is a single command:

./flowtree/rebuild.sh --agents

This rebuilds nothing if the jars are already compiled and the images are already built — Docker's layer cache means it completes in seconds. To skip the Maven build entirely when you have not changed Java code:

docker compose -f flowtree/controller/docker-compose.yml up -d
docker compose -f flowtree/agent/docker-compose.yml up -d

Optional: launch on login (macOS)

A launchd plist for ar-manager is provided at tools/mcp/manager/com.almostrealism.ar-manager.plist. For the full stack, a simple approach is a login item that runs ./flowtree/rebuild.sh --agents from the repository root.

Verify funnel is still active after reboot: 
tailscale funnel status
curl https://my-studio.taild1234.ts.net/_health
If the health check fails but Tailscale is up, the Docker containers stopped — run rebuild.sh to bring them back. The funnel itself does not need to be re-configured.

Troubleshooting

SymptomLikely causeFix
Agents log "connection refused" to controller Controller container is not running or port 7766 is firewalled Run docker compose ... ps; check FLOWTREE_ROOT_HOST in agent .env
/_health returns 502 from public URL ar-manager container stopped after reboot ./flowtree/rebuild.sh
Jobs submitted but agents never pick them up No agents connected yet, or peer connections not established Wait ~10 s after agent start; check logs for "Established connection"
curl: (60) SSL certificate problem HTTPS not enabled in Tailscale admin console Enable under DNS → HTTPS Certificates
tailscale funnel command not found Tailscale CLI too old or not in PATH Use /Applications/Tailscale.app/Contents/MacOS/Tailscale on macOS
ar-manager returns 401 on all requests Token file missing or token value copied incorrectly Check /Users/Shared/flowtree/manager/manager-tokens.json; regenerate with generate-token.sh
Two agents modify the same repository simultaneously Workspace directory is on a shared mount; lock contention The .flowtree.lock file inside each repo directory serialises access. Ensure agent workspace paths are not accidentally shared between different workstreams.
