Process Optimization System

Understanding Expression Tree Isolation and Kernel Boundaries

Home Overview The Problem Architecture Strategies Isolation Patterns Debugging Best Practices

Overview

The Process optimization system is a hierarchical framework for transforming computational work trees to improve execution efficiency. It automatically decides where to break up large expression trees into independently compiled kernels.

Critical Understanding: This system exists because without it, nested computations create expression trees that grow exponentially, causing compilation timeouts, stack overflows, and memory exhaustion.

Expression Embedding

When operations are nested, child expressions get embedded in parent expressions, creating a single large tree that must be compiled together.

Process Isolation

Breaking the tree into independent units that compile as separate kernels, with results passed through memory.

Optimization Strategies

Pluggable algorithms that analyze tree structure and decide when isolation improves performance.

Context Propagation

Information about parallelism, hardware requirements, and optimization state flows through the tree during optimization.

The Expression Embedding Problem

Without Isolation

When processes are not isolated, their expressions get embedded in parent expressions:

add(multiply(a, b), multiply(c, d)) ├── multiply expression (embedded in add) │ ├── a.getValueAt(i) (embedded) │ └── b.getValueAt(i) (embedded) └── multiply expression (embedded in add) ├── c.getValueAt(i) (embedded) └── d.getValueAt(i) (embedded)

This creates a single large expression tree that:

  • Takes exponentially longer to compile with depth
  • May overflow the call stack during compilation
  • Cannot be partially cached or reused

With Isolation

When processes are isolated, they compile as separate kernels:

add(isolate(multiply(a, b)), isolate(multiply(c, d))) ├── [Kernel 1: multiply(a, b)] → temp1 ├── [Kernel 2: multiply(c, d)] → temp2 └── [Kernel 3: add(temp1, temp2)]

Each kernel compiles independently, enabling:

  • Bounded compilation time per kernel
  • Kernel caching and reuse
  • Parallel compilation (future optimization)

Key Insight: The time is NOT spent in kernel execution, but in expression tree evaluation during the compilation phase. Isolation breaks this exponential growth.

Core Architecture

Process Hierarchy

Process<P, T> ├── optimize() - Entry point for optimization ├── isolate() - Wraps for independent execution ├── isIsolationTarget() - Determines if isolation needed └── getChildren() - Returns child processes ParallelProcess<P, T> extends Process ├── getParallelism() - Returns parallelism count ├── optimize(context, child) - Optimizes individual child └── isUniform() - Checks child parallelism equality ComputableParallelProcess<P, T> extends ParallelProcess ├── createContext() - Returns ComputableProcessContext └── isIsolationTarget() - Checks compute requirements

Context Hierarchy

ProcessContext ├── getOptimizationStrategy() - Returns current strategy └── getDepth() - Returns nesting level (0 = root) ParallelProcessContext extends ProcessContext ├── parallelism - Work items at current level ├── aggregationCount - Inputs per output └── fixed - Compile-time determined ComputableProcessContext extends ParallelProcessContext └── requirements - List<ComputeRequirement>

Optimization Flow

process.optimize() Entry Point
→
Create Context ParallelProcessContext
→
Optimize Children Recursive
→
Check Isolation isIsolationTarget()
→
Apply Strategy Final Decision
Optimization Algorithm: 
1. User calls process.optimize() or process.optimize(context)
   ↓
2. ParallelProcess.optimize(context) is invoked
   ├── If no children: return this (unchanged)
   ├── Create ParallelProcessContext from parent context
   ├── For each child:
   │   ├── Recursively call child.optimize(context)
   │   ├── Call optimize(context, optimizedChild) to decide isolation
   │   └── Result is either child or child.isolate()
   └── Delegate to context.getOptimizationStrategy().optimize(...)
   ↓
3. Strategy analyzes parallelism, memory, and tree structure
   ├── Extract parallelism and output size from each child
   ├── Calculate current score vs. isolated score
   ├── Apply strategy-specific rules
   └── Return optimized process (children isolated or not)

How IsolatedProcess Breaks Embedding

The Key Mechanism

IsolatedProcess does NOT implement TraversableExpression. When a parent's getValueAt() checks if a child can provide expressions:

if (child instanceof TraversableExpression) {
    // EMBEDDED: calls child.getValueAt(index)
    return ((TraversableExpression) child).getValueAt(index);
} else {
    // ISOLATED: returns null, forces separate kernel
    return null;
}

When getValueAt() returns null, the parent cannot embed the child, forcing a compilation boundary.

Optimization Strategies

Strategy Interface

public interface ProcessOptimizationStrategy {
    /**
     * Optimize a process tree.
     * @return Optimized process, or null if strategy doesn't apply
     */
    <P extends Process<?, ?>, T> Process<P, T> optimize(
        ProcessContext ctx,
        Process<P, T> parent,
        Collection<P> children,
        Function<Collection<P>, Stream<P>> childProcessor);
}

ParallelismTargetOptimization (Default)

Default Strategy - Threshold-Based Isolation

Makes isolation decisions based on parallelism counts and memory costs.

Parameter Default Value Purpose
minCount 256 (2^8) Minimum parallelism to consider
targetCount 131072 (2^17) Target threshold for decisions
maxCount 1048576 (2^20) Maximum before skip isolation
Decision Logic
  1. If ≤1 child OR total parallelism equals parent's → SKIP
  2. If max child > maxCount (1M) → SKIP (avoid fragmentation)
  3. If enableNarrowMax AND max child > targetCount AND context >= minCount → SKIP
  4. Calculate scores: parallelismValue - memoryCost
  5. If isolatedScore > currentScore → ISOLATE

TraversableDepthTargetOptimization

Depth-Based Isolation

Prevents expression trees from becoming too deep (which causes stack overflow).

// Constructor takes depth limit (default: 8)
new TraversableDepthTargetOptimization(8);

// Decision: If any child's depth > limit → ISOLATE

When to Use:

  • Deep expression trees from nested operations
  • Operations that compose many TraversableExpressions
  • When seeing stack overflow during compilation

AggregationDepthTargetOptimization

Aggregation-Based Isolation

Flattens deep aggregation trees (reductions, convolutions, etc.).

Threshold Value Purpose
AGGREGATION_THRESHOLD 64 Minimum aggregation count
PARALLELISM_THRESHOLD 128 Minimum child parallelism

Use Cases:

  • Large convolution operations with many input channels
  • Deep reduction trees
  • Operations with high fan-in

CascadingOptimizationStrategy

Chain Multiple Strategies

Chains multiple strategies; first successful result wins.

ProcessOptimizationStrategy strategy = new CascadingOptimizationStrategy(
    new TraversableDepthTargetOptimization(8),    // Check depth first
    new AggregationDepthTargetOptimization(12),   // Then aggregation
    new ParallelismDiversityOptimization(),       // Then diversity
    new ParallelismTargetOptimization()           // Finally, default
);
Design Principle: More specific strategies first (depth, aggregation), more general strategies last (parallelism target). Each strategy returns null if not applicable.

Strategy Selection Guide

Scenario Recommended Strategy Why
General use ParallelismTargetOptimization Balanced default behavior
Deep expressions TraversableDepthTargetOptimization(8) Prevents stack overflow
Large reductions AggregationDepthTargetOptimization(12) Handles high fan-in
Mixed workloads CascadingOptimizationStrategy Combines multiple heuristics
ML gradient computation Cascading with depth=4 Gradients create deep trees

Isolation Patterns

Pattern 1: Chain Breaking

Problem: Long chains of operations create deep expression trees
// Deep chain without isolation
result = op1(op2(op3(op4(op5(input)))));
// Expression depth: 5, grows exponentially with nesting
Solution: Insert isolation points in the chain
// With strategic isolation
temp1 = op1(op2(input)).optimize().get().evaluate();
temp2 = op3(op4(op5(p(temp1)))).optimize().get().evaluate();

Automatic: Use TraversableDepthTargetOptimization(4)

Pattern 2: Multi-Input Aggregation

Problem: Operations with many inputs embed all input expressions
// Concatenation of many tensors - all expressions embedded
result = concat(t1, t2, t3, t4, t5, t6, t7, t8);
Solution: Ensure inputs are isolated or pre-computed
// Compute heavy inputs first
List<PackedCollection> precomputed = heavyOps.stream()
    .map(op -> op.optimize().get().evaluate())
    .collect(toList());

// Concat from memory references
result = concat(precomputed.stream().map(this::p).toArray(Producer[]::new));

Pattern 3: Gradient Computation

Problem: Automatic differentiation creates exponentially growing trees
// Forward: y = f(x)
// Backward: dy/dx = chain rule across all operations
CollectionProducer gradient = output.delta(input);
// Expression tree includes all intermediate derivatives
Solution: Use TransitiveDeltaExpressionComputation
public class MyComputation extends TransitiveDeltaExpressionComputation {
    @Override
    public CollectionProducer delta(Producer<?> target) {
        // Create isolated delta computation
        List<Process<?, ?>> deltas = computeInputDeltas(target);
        return generate(deltas).reshape(outputShape.append(targetShape));
    }
}

Key Insight: Each transitive delta computation creates an isolated boundary.

Anti-Pattern 1: Over-Isolation

BAD: Isolating tiny operations
for (int i = 0; i < 1000; i++) {
    PackedCollection result = add(a, b).optimize().get().evaluate();
    // 1000 kernel launches instead of 1!
}

Fix: Batch operations or raise isolation thresholds.

Anti-Pattern 2: Under-Isolation

BAD: No isolation on deep nested computation
OperationList ops = buildDeepComputationGraph();
Runnable r = ops.get();  // Missing optimize()!
// Potential stack overflow or timeout

Fix: Always call optimize() before get() for complex computations.

Anti-Pattern 3: Hacking getValueAt()

BAD: Returning null from getValueAt() to force isolation
@Override
public Expression<Double> getValueAt(Expression index) {
    if (shouldForceIsolation()) {
        return null;  // WRONG! Violates architecture
    }
    return computeValue(index);
}

Fix: Use isIsolationTarget() and proper optimization flow.

Debugging Isolation Issues

Symptoms of Missing Isolation

  • Test timeouts (60+ seconds for simple operations)
  • Stack overflow during compilation
  • OutOfMemoryError during expression evaluation
  • Massive expression trees in stack traces
  • Profile showing time in "kernelSeries" evaluation stages

Check If Isolation Is Happening

// Add isolation logging
ParallelProcess.isolationFlags.add(p -> {
    System.out.println("[CHECK] " + p.getClass().getSimpleName() +
        " isIsolated=" + (p instanceof IsolatedProcess));
    return true;
});

Trace Optimization Decisions

// Log when optimization runs
ProcessOptimizationStrategy.listeners.add(parent -> {
    System.out.println("[OPT] Processing: " + parent);
});

// Enable verbose logging
ComputableParallelProcess.enableOptimizationLog = true;

Verify Expression Depth

// Override in your process
@Override
public boolean isIsolationTarget(ProcessContext ctx) {
    int depth = calculateDepth(this);
    System.out.println("Depth of " + this + ": " + depth);
    return depth > 8 || super.isIsolationTarget(ctx);
}

Profile Kernel Compilation

// Measure optimization and compilation separately
long start = System.nanoTime();
OperationList optimized = (OperationList) ops.optimize();
long optimizeTime = System.nanoTime() - start;

start = System.nanoTime();
Runnable r = optimized.get();
long compileTime = System.nanoTime() - start;

System.out.println("Optimize: " + (optimizeTime / 1e6) + "ms");
System.out.println("Compile: " + (compileTime / 1e6) + "ms");

Using Profile Analysis

Profile XML Analysis: Use the ar-profile-analyzer MCP tool to identify slow operations:
  • find_slowest - Find the N slowest operations
  • get_timing_breakdown - See compile vs run time
  • list_children - Drill into operation hierarchy

Look for operations where compile time >> run time, indicating expression tree issues.

Best Practices

1. Always Call optimize() Before get()

// CORRECT
OperationList ops = model.getForward().push(input);
ops = (OperationList) ops.optimize();  // Critical!
Runnable r = ops.get();

2. Use Cascading Strategies for Complex Workloads

ProcessContextBase.setDefaultOptimizationStrategy(
    new CascadingOptimizationStrategy(
        new TraversableDepthTargetOptimization(8),
        new AggregationDepthTargetOptimization(12),
        new ParallelismTargetOptimization()
    )
);

3. Use TransitiveDeltaExpressionComputation for Gradients

For operations that participate in backpropagation, extend TransitiveDeltaExpressionComputation to get automatic isolation of gradient computations.

4. Monitor Compilation Times

If compilation takes >1 second for a simple operation, you likely have an isolation issue.

5. Profile First, Optimize Second

Use profile XML analysis to identify where time is actually spent before adjusting isolation strategies.

Performance Trade-offs

Scenario Memory Compilation Execution
No isolation Single output buffer One large, slow compilation One kernel launch (if it compiles)
Full isolation Buffer per operation Many small, fast compilations Many kernel launches
Strategic isolation Minimal intermediate buffers Balanced compilation time Optimal kernel granularity
