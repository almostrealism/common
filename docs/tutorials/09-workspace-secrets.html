Tutorial: Workspace Secrets for Coding Agents

Home Tutorials FlowTree Stack Setup FlowTree Module

Introduction

The FlowTree controller ships with a workspace-scoped secret manager that lets a ClaudeCodeJob agent pull credentials it needs (AWS keys, API tokens, database passwords, etc.) without ever seeing the raw values in its conversation context. The agent fetches a secret, renders it into a file on disk through a template, and the rendered content is consumed by another process — never echoed back to the model.

This tutorial walks through the full lifecycle:

  • Declaring a secret in workstreams.yaml so the controller knows about it
  • Storing the payload (either by pre-creating the JSON file or using the admin HTTP API)
  • Telling a coding agent — running inside a ClaudeCodeJob — that the secret is available and how to use it
  • Two end-to-end illustrations: an AWS CLI credentials file and a generic JSON file with username/password fields

Prerequisites

  • A working FlowTree stack — see Tutorial 08
  • Admin shell access to the controller host (where workstreams.yaml lives)
  • The shared secret used by the controller and ar-manager (AR_MANAGER_SHARED_SECRET) — required for admin-API calls and for the agent's HMAC token
  • The Slack workspace ID (a value starting with T) that owns the workstream you intend to use the secret from

Docker path mapping — read this first

The reference stack runs the controller inside Docker. Every path in workstreams.yaml is interpreted from inside the controller container, not from your host. If you write a host path like /Users/Shared/flowtree/secrets/aws-prod.json into the YAML, the controller will look for that path inside the container, not find it, and log Secret file not found.

The shipped flowtree/controller/docker-compose.yml mounts two host directories into the controller and ar-manager containers:

Host path (where you edit files) Container path (what YAML must reference) Mode
/Users/Shared/flowtree/controller/ /config/ rw — workstreams.yaml, slack-tokens.json
/Users/Shared/flowtree/secrets/ /secrets/ ro by default — secret JSON payload files
The secrets mount is read-only by default. Option A (pre-create the payload file on the host) works out of the box. Option B (PUT via the admin HTTP API) requires the operator to drop the :ro flag on the secrets mount first — see Step 2.

Throughout the rest of this tutorial, paths inside workstreams.yaml are always container paths (rooted at /config or /secrets), and shell commands you run on the host always use host paths (rooted at /Users/Shared/flowtree/...). If you are running the controller without Docker — directly on the host JVM — use plain host paths everywhere.

How It Works

HOST CONTAINER (flowtree-controller) ─────────────────────────── ─────────────────────────────── /Users/Shared/flowtree/ ┌──────────────────────────────┐ controller/workstreams.yaml ──▶│ /config/workstreams.yaml │ ① Declare each secret: │ slackWorkspaces: │ name + container path │ - workspaceId: T0123… │ │ secrets: │ │ - name: aws-prod │ │ file: │ │ /secrets/aws.json│ └──────────────┬───────────────┘ │ /Users/Shared/flowtree/ ▼ secrets/aws.json (0600) ───▶ ┌──────────────────────────────┐ ② Payload on disk: │ /secrets/aws.json (ro) │ { "access_key_id":"…", │ │ "secret_access_key":"…" } └──────────────┬───────────────┘ │ workspace_secret_render_file( │ template, output_path) ▼ ┌──────────────────────────────┐ ③ Agent renders into a file │ ClaudeCodeJob agent │ — payload never crosses │ (Claude Code + ar-manager) │ back to the model. └──────────────────────────────┘

Three actors, three roles:

  • The operator declares secrets in workstreams.yaml and seeds their payloads.
  • The controller indexes secrets by Slack workspace ID, enforces 0600 file permissions, and serves the /api/secrets/* endpoints.
  • The agent calls the workspace_secret_list_names and workspace_secret_render_file MCP tools exposed by ar-manager. The rendered file lands on disk in the agent's workspace; the values themselves never flow back through the MCP channel.
Workspace scope, workstream access. Secrets are owned by a Slack workspace. Every workstream that lives in that workspace can read every secret declared on it. If you need finer-grained scoping, declare separate workspaces for separate trust boundaries.

1 Declare the secret in workstreams.yaml

Edit workstreams.yaml on the host at /Users/Shared/flowtree/controller/workstreams.yaml. Add a secrets: list under the slackWorkspaces entry that owns it. Each entry is a (name, file) pair. The name is the handle the agent will reference; the file is the path inside the container — for the reference stack that means /secrets/<something>.json, which maps to /Users/Shared/flowtree/secrets/<something>.json on the host.

slackWorkspaces:
  - workspaceId: "T0123456789"
    name: "primary-team"
    tokensFile: "/config/slack-tokens.json"      # container path → host: …/controller/slack-tokens.json
    secrets:
      - name: "aws-prod"
        file: "/secrets/aws-prod.json"           # container path → host: …/secrets/aws-prod.json
      - name: "demo-app-credentials"
        file: "/secrets/demo-app-credentials.json"
Do not use host paths here. Writing /Users/Shared/flowtree/secrets/aws-prod.json in workstreams.yaml will look correct — it's the right path on the host — but the controller resolves it from inside the container, where that directory does not exist. The startup log will read Secret file not found and the agent will get 404 Not Found on render. Always use /config/… and /secrets/….
Running without Docker? If you start the controller JVM directly on the host (no Docker), use plain host paths everywhere in workstreams.yaml — there's no path translation in that case.
Naming rules. Secret names must be unique within a workspace and must be URL-safe — lowercase letters, digits, and hyphens are the safe set. They appear verbatim in URLs the agent calls.

Reload the controller so the new declarations are picked up:

docker compose -f flowtree/controller/docker-compose.yml restart flowtree-controller

# In the controller log you should see:
# Loaded 2 secret(s) across 1 workspace(s)

If the controller logs Secret file not found that is fine for now — we will create the file in step 2. The declaration alone is enough for the next step to work.

2 Store the secret payload

The payload is a flat JSON map of string keys to string values. There are two equivalent ways to seed it.

Option A — pre-create the file on the host

Quickest for first-time setup, and the only path that works without changing the default Docker mounts. The controller does not need to be running. Create the file on the host in /Users/Shared/flowtree/secrets/; the container will see it at /secrets/ once the controller next starts.

sudo mkdir -p /Users/Shared/flowtree/secrets
sudo chown "$(whoami)" /Users/Shared/flowtree/secrets

# Host path. Maps to /secrets/aws-prod.json inside the container,
# which is what workstreams.yaml references.
cat > /Users/Shared/flowtree/secrets/aws-prod.json <<'EOF'
{
  "access_key_id": "AKIAIOSFODNN7EXAMPLE",
  "secret_access_key": "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY",
  "region": "us-east-1"
}
EOF

chmod 600 /Users/Shared/flowtree/secrets/aws-prod.json
The controller will warn at startup if the file's POSIX permissions are wider than 0600 (group- or world-readable). Fix it before continuing — a leaked credential file on a shared host is a real incident.

Option B — use the admin HTTP API

Use this when you want to update a secret in place without shelling into the host filesystem. The endpoint is PUT /api/secrets/{name}?workspace_id=..., authenticated with the controller's shared secret as a Bearer token. The endpoint refuses to create entries that aren't already declared in workstreams.yaml; the YAML declaration in step 1 is a prerequisite.

The PUT API needs a writable secrets mount. The reference docker-compose.yml mounts /Users/Shared/flowtree/secrets as /secrets:ro — read-only — for both the controller and ar-manager. With that default, PUT will fail with 500 Internal Server Error when it tries to write the payload file. To use Option B, edit flowtree/controller/docker-compose.yml and remove the :ro suffix on the flowtree-controller service's secrets volume (keep ar-manager's mount read-only — it never needs to write), then restart. If you would rather not loosen the mount, stick to Option A. 
# Run from any host that can reach the controller's HTTP port (7780).
# The shared-secret file lives on the host beside the manager config.
SHARED_SECRET=$(cat /Users/Shared/flowtree/manager/shared-secret)
WORKSPACE_ID="T0123456789"

curl -X PUT "http://localhost:7780/api/secrets/aws-prod?workspace_id=${WORKSPACE_ID}" \
    -H "Authorization: Bearer ${SHARED_SECRET}" \
    -H "Content-Type: application/json" \
    --data-binary @- <<'EOF'
{
  "access_key_id": "AKIAIOSFODNN7EXAMPLE",
  "secret_access_key": "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY",
  "region": "us-east-1"
}
EOF

The controller writes the JSON atomically (temp file → chmod 0600 → atomic rename) so there is no window where a partial or world-readable file is observable. The write happens to the container path (/secrets/aws-prod.json), which the bind mount projects back onto the host at /Users/Shared/flowtree/secrets/aws-prod.json.

Verify the secret is visible

# Admin token can list every secret in the workspace.
curl -H "Authorization: Bearer ${SHARED_SECRET}" \
    "http://localhost:7780/api/secrets?workspace_id=${WORKSPACE_ID}"
# {"names":["aws-prod","demo-app-credentials"]}

3 Tell the coding agent the secret exists

A ClaudeCodeJob agent connects to ar-manager as its primary MCP server. Two tools matter here:

ToolWhat it does
workspace_secret_list_names Returns the names of secrets available to the agent's workstream. No payloads. Useful as a discovery step.
workspace_secret_render_file Fetches the named payload, substitutes {{key}} placeholders in a template, and writes the rendered content to a file path with the requested chmod. Returns the output path on success — never the rendered content.

The agent doesn't need any extra configuration to call these tools — they are part of the standard ar-manager toolset delivered to every job. What the agent does need is a clear instruction in the job prompt that tells it:

  1. The exact secret name to fetch.
  2. The shape of the payload (which keys exist).
  3. The template to use and the destination file path.
Why be explicit about the keys? The render tool is strict — every {{key}} in the template must exist in the payload, or it returns an error and writes no file. Telling the agent which keys are present avoids a discovery round-trip and a guessing game.

The next two sections show concrete prompts. Submit them via workstream_submit_task, the FlowTree Slack /agent command, or any other path that lands a ClaudeCodeJob on an agent.

Illustration 1 — Populate ~/.aws/credentials

Goal: the agent should be able to run aws s3 ls and similar CLI commands using a long-lived production access key, without that key ever appearing in a model response, log line, or commit.

Payload schema

The JSON file (aws-prod.json) holds three keys. They map directly onto fields in the standard AWS credentials INI file.

{
  "access_key_id": "AKIA…",
  "secret_access_key": "wJal…",
  "region": "us-east-1"
}

Prompt to give the agent

Submit this as the task description. Keep it specific about which secret to use and what file to render — the agent will follow it verbatim.

You have access to a workspace secret named aws-prod that contains AWS credentials for the production account. The payload has three keys: access_key_id, secret_access_key, and region. Before doing any AWS work, render the secret into ~/.aws/credentials using the standard INI layout: [default] aws_access_key_id = {{access_key_id}} aws_secret_access_key = {{secret_access_key}} region = {{region}} Use workspace_secret_render_file with mode "0600". Do NOT read the rendered file back, do NOT echo any value from the payload, and do NOT include credential values in any commit, PR description, or log line. Once the file is in place, run "aws s3 ls" to verify the credentials work, then proceed with the actual task: …

What the agent will do

# 1. (Optional) Discover what's available
workspace_secret_list_names(workstream_id="ws-abc-123")
# → {"ok": true, "names": ["aws-prod", …]}

# 2. Render the credentials file
workspace_secret_render_file(
    workstream_id="ws-abc-123",
    secret_name="aws-prod",
    template="""[default]
aws_access_key_id = {{access_key_id}}
aws_secret_access_key = {{secret_access_key}}
region = {{region}}
""",
    output_path="~/.aws/credentials",
    mode="0600",
)
# → {"ok": true, "output_path": "/home/agent/.aws/credentials"}

# 3. Use the AWS CLI normally — no further secret handling needed
aws s3 ls

Illustration 2 — Render a JSON config with username/password

Goal: a tool the agent runs (a database CLI, a custom deployment script, a third-party service client) reads its credentials from a JSON file with separate username and password fields. The agent must produce that file from a workspace secret without ever seeing the values.

Payload schema

The JSON file (demo-app-credentials.json) holds whatever flat keys you need. Here we have two:

{
  "username": "deployer-bot",
  "password": "5n0wF14k3-W1nt3r-2026"
}

Prompt to give the agent

You have access to a workspace secret named demo-app-credentials that contains login details for the demo app's deployment endpoint. The payload has two keys: username and password. The deployment script reads its credentials from ./config/demo-app.json, which expects the following shape: { "username": "{{username}}", "password": "{{password}}" } Render the secret into that path using workspace_secret_render_file with mode "0600". Do NOT read the file back, do NOT print either value, and do NOT commit the rendered file (it is git-ignored). Once the file is in place, run "./scripts/deploy.sh" and report the exit status only.

What the agent will do

workspace_secret_render_file(
    workstream_id="ws-abc-123",
    secret_name="demo-app-credentials",
    template="""{
  "username": "{{username}}",
  "password": "{{password}}"
}
""",
    output_path="./config/demo-app.json",
    mode="0600",
)
# → {"ok": true, "output_path": "/workspace/project/config/demo-app.json"}
Embedding values inside JSON strings is fine — the renderer is a literal text substitution. As long as your payload values don't contain unescaped quotes or backslashes, the resulting JSON is valid. If a value might contain JSON-special characters, store it pre-escaped in the payload, or use a generated config format (YAML, TOML, INI) where escaping is less fragile.

Security Rules — Read These

The render tool is built so that secret values never enter the agent's context, but those guarantees only hold if the operator and the agent both follow a few rules. The tools/mcp/CLAUDE.md guidelines spell these out for agents; the equivalents for operators are:

RuleReason
Never instruct the agent to read the rendered file back (cat, Read, etc.). That would reintroduce the values into the model's context — defeating the purpose.
Never include payload values in a job prompt or in any other field that the agent will see. The whole point is that the agent gets a handle (the secret name), not the value.
Add the rendered file path to .gitignore in any repo the agent touches. Prevents an accidental commit of rendered credentials. The rendered file should be treated as ephemeral.
Keep the on-disk payload file at 0600; rotate the credential when the host is shared. The controller warns when permissions are wider than 0600. Treat that warning as a hard failure.
Never log payload values from custom job tooling. The audit log already records secret access (workstream ID, secret name, output path) but deliberately omits payload contents. Match that discipline elsewhere.
Audit log location. Every workspace_secret_render_file call appears in the ar-manager audit log with the workstream ID, secret name, and output path — but never the template or values. Tail this log when investigating suspected secret access: docker compose -f flowtree/controller/docker-compose.yml logs ar-manager | grep secret_.

Troubleshooting

Symptom Likely cause Fix
Agent reports Secret not declared in config Name is missing from the secrets: list in workstreams.yaml Add the entry, restart the controller, retry
Template references unknown keys: […] Template uses a {{key}} the payload doesn't have Either store that key in the payload (PUT the secret again with the missing field) or remove the placeholder from the template
Shared secret not configured on ar-manager AR_MANAGER_SHARED_SECRET not set on ar-manager Set it in the controller stack docker-compose.yml and restart ar-manager
Controller log says Secret file not found Either (a) the declaration exists but no payload on disk yet, or (b) the file: path in workstreams.yaml is a host path (/Users/Shared/...) instead of the container path (/secrets/...) Confirm the file exists on the host at /Users/Shared/flowtree/secrets/<name>.json, and that workstreams.yaml references it as /secrets/<name>.json. See Docker path mapping.
PUT to /api/secrets/<name> returns 500 with Failed to write secret file The secrets volume is mounted :ro in docker-compose.yml Remove :ro on the flowtree-controller secrets volume and restart, or use Option A (pre-create on the host) instead
Secret file has unsafe permissions Payload file is group- or world-readable chmod 600 <file>
Agent can list names but render fails with 403 Workstream HMAC token expired or workstream ID in the request mismatches the token Resubmit the job — tokens are minted fresh per job and last 12 hours
