Overview

FlowTree is a distributed processing framework built on a decentralized peer-to-peer network. Each node in the network can accept, execute, and relay jobs to other nodes, enabling work to be distributed across an arbitrary number of machines without a central coordinator.

A FlowTree network consists of Servers, each hosting a NodeGroup containing one or more Nodes. Jobs enter the system through a NodeGroup, are queued on child Nodes, and are either executed locally or relayed to peer Nodes on other Servers.

Two Networking Layers

FlowTree has two distinct connection types. Confusing them is a common source of bugs.

Server Connections (NodeProxy)

Socket-level links between two Servers. Stored in NodeGroup.servers. Created when an agent Server connects to the controller. Carry Message objects (tasks, connection requests, job data).

Peer Connections (Connection)

Logical links between individual Nodes on different Servers. Stored in each Node.peers. Wrap a NodeProxy and target a specific remote Node by ID. This is how jobs actually move between Nodes.

Controller Server Agent Server NodeGroup NodeGroup servers: [NodeProxy] <-TCP-> servers: [NodeProxy] Node 0 (relay) Node 0 peers: [Connection] <---> peers: [Connection]

How Peer Connections Are Established

Peer connections are not configured manually. Each Node's activity thread periodically requests connections:

  1. Node checks if it has fewer peers than maxPeers
  2. Calls parent.getConnection(this.id)
  3. NodeGroup picks a random entry from servers and sends a ConnectionRequest message
  4. Remote NodeGroup receives the request, creates a Connection, responds with ConnectionConfirmation
  5. Both sides now have a Connection in their peers set

Job Relay Mechanism

Each Node runs two independent threads:

Worker Thread

Pulls jobs from the queue and executes them. On relay Nodes (role:relay), the worker thread is never started — jobs remain in the queue exclusively for the relay loop.

Activity Thread

Manages peer connection establishment and relays excess jobs to peers. When the queue has jobs above minJobs, selects a random peer and calls Connection.sendJob(). This is the only mechanism that moves jobs between Nodes on different Servers.

Critical invariant: Jobs are never filtered at the queue level. Any Node accepts any job into its queue via addJob(). On relay Nodes, the worker thread is never started, so jobs remain in the queue for the relay loop. On execution Nodes, the worker pulls and runs jobs directly.

Label-Based Routing

Node Labels

Each Node has a label map set via properties (nodes.labels.<key>=<value>), environment variable (FLOWTREE_NODE_LABELS), or auto-detection (the platform label).

Job Requirements

Each Job can declare required labels via getRequiredLabels(). A Node satisfies requirements if every required key-value pair exists in the Node's labels.

The role:relay Label

A Node with role:relay never executes jobs. satisfies() always returns false regardless of the job's requirements. The controller uses this to create a queue-only Node that warehouses jobs for relay to agent Nodes.

Design principle: The relay Node is not a special mechanism. It uses exactly the same relay infrastructure as any other Node. The only difference is that every job is rejected by the worker, keeping the queue full and triggering the relay loop.

Key Classes

Node

Processes jobs, manages peer connections, relays excess work. Contains worker thread (execution) and activity thread (connections + relay).

NodeGroup

Collection of child Nodes. Manages server connections (NodeProxy list). Distributes jobs from factories to child Nodes. Brokers peer connection requests.

Connection

Peer-to-peer link wrapping a NodeProxy. Targets a specific remote Node by ID. Used by the relay loop to send jobs between Nodes on different Servers.

NodeProxy

Socket-level server connection. Carries Message objects between Servers. Multiple Connections can share one NodeProxy.

Message

Wire-level protocol unit. Types include Job (relay a job), Task (distribute a factory), ConnectionRequest, and ConnectionConfirmation.

Server

Entry point. Hosts a NodeGroup, accepts incoming socket connections, manages thread groups.

Coding Agent Integration

FlowTree integrates with Claude Code to execute AI coding prompts as distributed jobs. The controller dispatches ClaudeCodeJob.Factory instances to connected agent nodes; each agent runs Claude Code in a headless subprocess (claude -p) and commits the resulting changes via GitManagedJob.

Inactivity Monitor and Restart Loop

Each Claude subprocess is supervised by a daemon ClaudeInactivityMonitor that destroys the process and its descendants when stdout has been silent for inactivityTimeoutMillis. The most common cause of indefinite hangs in autonomous agent sessions is a shell polling loop where the wait condition never becomes true — for example, pgrep -f <pattern> matching its own command line, or curl-against-an-invented endpoint inside a bash until loop. The monitor catches both classes of failure mechanically.

When the monitor fires, ClaudeCodeJob relaunches Claude up to maxInactivityRestarts times. Each relaunch carries a prompt preamble (## !! SESSION RESTARTED -- INACTIVITY TIMEOUT !!) that explains the polling-loop failure mode and points the agent to the MCP get_*_status tools as the correct way to wait on a managed run. Prior progress on the branch is preserved in git, so the restart picks up where the killed attempt left off rather than starting from scratch.

Deduplication Check

Agent sessions routinely re-implement functionality that already exists elsewhere in the codebase — renamed, slightly generalised, or placed in a different class. The deduplication check runs after Claude Code finishes and before changes are committed. It scans the diff for new Java method declarations and submits an aggressive follow-up prompt that instructs a second agent to find and eliminate any duplicates.

DEDUP_LOCAL ("local")

Runs an inline Claude Code session with the deduplication prompt before the commit is finalised. The dedup edits land in the same working tree and are committed together with the original work. No extra jobs are created. Recommended while iterating.

DEDUP_SPAWN ("spawn")

Posts a follow-up ClaudeCodeJob to the same workstream via POST /api/submit after the current commit. Fire-and-forget; requires a workstream URL. Useful in production pipelines where the dedup pass should be independently observable.

Infinite loop safety. DEDUP_LOCAL calls executeSingleRun() directly. This never re-invokes validateChanges() and never copies deduplicationMode to the inner session, so the check runs exactly once.

Via ar-manager MCP

The workstream_submit_task MCP tool exposes deduplication_mode:

workstream_submit_task(
    prompt="Implement feature X",
    workstream_id="ws-common",
    deduplication_mode="local"
)

Common Pitfalls

Do not bypass the relay mechanism. Do not send jobs directly to a specific server index. Do not create parallel relay paths outside the activity thread. Do not attempt to relay from the worker thread. The activity thread's relay loop is the only correct way to move jobs between Nodes on different Servers.
Relay depends on peer connections. Jobs will not move to remote Servers until peer Connections are established. This happens automatically but takes a few seconds after the initial server connection. If jobs appear stuck, check that nodes.peers.max is at least 1.
Server connections are not peer connections. NodeGroup.servers (NodeProxy) and Node.peers (Connection) are completely separate. Code that confuses them will silently fail to route jobs.
