Instruction Set Caching

Signature-Based Kernel Compilation Caching with Argument Substitution

Home Hardware Module Heap Overview Architecture Signatures Components Substitution Lifecycle Configuration

Overview

The instruction set caching system avoids redundant compilation of hardware kernels by caching compiled InstructionSet instances and reusing them across operations that share the same structural signature. Compiling a computation to native code (JNI, OpenCL, Metal) is expensive. Many operations in a computation graph share identical structure but differ only in their data arguments. The caching system exploits this by computing a structural signature, caching the compiled kernel, and substituting arguments when reusing.

Structural Signatures

Each operation is assigned an MD5 signature based on its type, shapes, precision, and input structure — not data values. Two operations with the same signature produce identical kernel code.

FrequencyCache

Compiled kernels are stored in a hybrid LFU/LRU cache (FrequencyCache) with capacity 500 and a frequency bias of 0.4. Evicted entries have their native resources automatically destroyed.

Argument Substitution

When a cached kernel is reused, ProcessArgumentMap maps the new operation's data arguments to the kernel's parameter positions, avoiding recompilation entirely.

Lazy Compilation

ScopeInstructionsManager compiles on first access via getOperator(). Subsequent calls return the cached operator instantly.

Two operations with the same signature produce identical kernel code. They differ only in which memory buffers they operate on.

Signature Generation

Stage 1: ProducerComputationBase.signature()

Recursively collects signatures from all input operations (skipping the first input, which is the destination). Combines the computation name, compute requirements, and all input signatures into an MD5 hash. If any input lacks a signature, the entire chain returns null and caching is disabled for this operation.

String combined = getName() + "/" + requirements + "|" +
        String.join(":", inputSignatures);
return Signature.md5(combined);
// Result: "a3f2b7c4e8d1..."  (32-char hex MD5)

Stage 2: ComputationScopeCompiler.signature()

Appends &distinct=N; to the base signature, where N is the number of distinct children in the Process tree. This distinguishes add(A, A) (distinct=1) from add(A, B) (distinct=2).

// Final format:
// a3f2b7c4e8d1...&distinct=2;
//
// - MD5 hash of structural identity
// - Number of distinct child processes

FrequencyCache<K, V>

Package: io.almostrealism.util

Hybrid LFU/LRU cache used by DefaultComputer to hold compiled instruction managers.

Eviction Score Formula

score = frequencyBias * (frequency / totalAccesses)
      + (1 - frequencyBias) * (1 - age)

// With frequencyBias = 0.4:
// score = 0.4 * normalizedFrequency + 0.6 * normalizedRecency
//
// Recency (60%) outweighs frequency (40%):
// recently-used kernels are preferred over historically popular ones.
MethodDescription
get(K key)Retrieves value, updates frequency and clock. Returns null if absent.
put(K key, V value)Inserts or updates mapping. Deduplicates values via reverse map. May trigger eviction.
computeIfAbsent(K key, Supplier<V>)Returns cached value or computes and caches if absent.
evict(K key)Explicitly removes entry and invokes eviction listener.
setEvictionListener(BiConsumer)Registers callback for eviction events (e.g., resource cleanup).
valuesByFrequency(IntPredicate)Streams values filtered and sorted by access frequency.
Value Deduplication: Multiple keys can map to the same CacheEntry via a reverse map. Frequency tracking is unified across all keys pointing to the same value. Eviction of a value removes all associated keys.

Argument Substitution

When a cached kernel is reused by a different operation instance, the compiled code is identical but the data arguments differ. ProcessArgumentMap enables this by mapping Process tree positions to scope arguments and supporting dynamic substitution.

ProcessArgumentMap

Package: org.almostrealism.hardware.arguments

FieldTypePurpose
argumentsByPositionMap<ProcessTreePositionKey, ArrayVariable>Maps tree positions to scope arguments
positionsForArgumentsMap<ArrayVariable, ProcessTreePositionKey>Reverse map: argument to tree position

Substitution Flow

When AcceleratedComputationOperation.load() detects shared instruction reuse (arguments are null after loading the operator):

1
Get the shared ScopeInstructionsManager and its scope metadata (inputs, arguments)
2
Call setupArguments() to copy the shared scope's input suppliers and argument list to this operation
3
Create a copy of the ProcessArgumentMap (with fresh, empty substitution map)
4
putSubstitutions() walks this operation's Process tree and records each node's Producer at the corresponding tree position
5
Set the substitution map as the evaluator. At kernel execution time, argument resolution uses the substituted Producers instead of the originals
// In AcceleratedComputationOperation.load():
ScopeInstructionsManager manager =
    (ScopeInstructionsManager) getInstructionSetManager();

// Copy scope metadata to this operation
setupArguments(manager.getScopeInputs(), manager.getScopeArguments());

// Create fresh substitution map
ProcessArgumentMap map = new ProcessArgumentMap(manager.getArgumentMap());
map.putSubstitutions((Process<?,?>) getComputation());

// Use substituted arguments for execution
setEvaluator(map);

Lifecycle

First Compilation (Cache Miss)

1
getInstructionSetManager() computes signature, calls DefaultComputer.getScopeInstructionsManager()
2
FrequencyCache.computeIfAbsent() misses: creates new ScopeInstructionsManager with lazy scope supplier
3
load() calls manager.getOperator(key), which invokes the scope supplier, compiles to InstructionSet, returns operator
4
Arguments are available from compilation; normal execution path

Subsequent Reuse (Cache Hit)

1
getInstructionSetManager() computes same signature, calls DefaultComputer.getScopeInstructionsManager()
2
FrequencyCache.computeIfAbsent() hits: returns existing ScopeInstructionsManager
3
load() calls manager.getOperator(key); operators is non-null so returns cached operator instantly — NO recompilation
4
Arguments are null (not from this operation's compilation); argument substitution via ProcessArgumentMap

Eviction

1
FrequencyCache exceeds capacity 500; computes eviction scores
2
Entry with lowest score is removed; eviction listener calls manager.destroy()
3
ScopeInstructionsManager.destroy() destroys the InstructionSet (releases native code) and runs all destroy listeners
4
If a lingering operation reference accesses the manager, the access listener restores it to the cache and getOperator() recompiles

Relationship to Heap

Heap and instruction caching are independent systems operating at different levels:

AspectHeapInstruction Caching
What it cachesMemory allocations (Bytes)Compiled kernels (InstructionSet)
ScopeThread-local, per-stageGlobal, per-signature
LifetimeScope exit (push/pop)Until eviction from FrequencyCache
Resource typeMemory buffersNative compiled code
Important: Heap.stage() destroys OperationAdapter wrappers via HeapDependencies but does not invalidate the instruction cache. A destroyed OperationAdapter triggers recompilation only if the InstructionSet within its ScopeInstructionsManager has also been independently destroyed (e.g., by cache eviction).

Common Pitfalls

Null Signatures Disable Caching: If any input in a computation tree does not implement Signature or returns null from signature(), the entire chain returns null and instruction caching is disabled. Every evaluation triggers a fresh compilation. Check whether signature() returns null for the computation.
Eviction Triggers Recompilation: When a ScopeInstructionsManager is evicted, its InstructionSet is destroyed. Operations still holding a reference will recompile on next use. This manifests as periodic compilation spikes in long-running applications.
Argument Substitution Requires Process: The ProcessArgumentMap.putSubstitutions() mechanism requires the computation to implement Process. Non-Process computations fall back to full recompilation with a log warning.
Access Listener Restores Evicted Entries: DefaultComputer creates managers with an access listener that calls instructionsCache.computeIfAbsent(signature, () -> mgr). This means using an evicted manager restores it to the cache, preventing repeated eviction of active kernels.
