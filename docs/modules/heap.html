Heap

Thread-Local Arena Allocator with Staged Scopes and Dependency Lifecycle Management

Home Hardware Module Overview Architecture API Reference Usage Patterns Integration Lifecycle

Overview

Heap is a thread-local arena allocator in the org.almostrealism.hardware.mem package. It pre-allocates a single large memory block and serves allocation requests by advancing a bump pointer, reducing allocation overhead from O(n) provider calls to a single allocation plus O(1) pointer bumps.

Arena Allocation

Pre-allocates a large memory block and suballocates via bump pointer advancement. Each allocation returns a zero-copy Bytes view into the backing block — no new memory provider calls are made.

Staged Scopes

Nested stages create independent allocation scopes. When a stage exits, all its allocations and tracked dependencies are automatically freed. Stages can be nested arbitrarily deep.

Dependency Lifecycle

Each stage tracks compiled operations, dependent operations, and temporary memory. When the stage is destroyed, all tracked resources are automatically cleaned up in a defined order.

Thread-Local Default

A thread-local default heap enables transparent integration. When active, PackedCollection.factory(), MemoryDataAdapter.init(), and other framework components automatically use heap-backed allocation.

Architecture

Memory Layout

+------------------------- Heap --------------------------+
|                                                          |
|  +------------ Root Stage (rootSize) ----------------+  |
|  |  [alloc1][alloc2][alloc3]...        [free space]   |  |
|  |  ^                           ^end                  |  |
|  |  HeapDependencies: ops, compiled, memory           |  |
|  +----------------------------------------------------+  |
|                                                          |
|  +------------ Pushed Stage (stageSize) --------------+  |
|  |  [alloc4][alloc5]...                [free space]   |  |
|  |  ^                     ^end                        |  |
|  |  HeapDependencies: ops, compiled, memory           |  |
|  +----------------------------------------------------+  |
|                                                          |
|  (additional pushed stages as needed)                    |
+----------------------------------------------------------+

The root stage is created at construction time and persists for the heap's lifetime. Additional stages are pushed and popped via push()/pop() (or Heap.stage()). Each stage is an independent bump allocator with its own backing Bytes block and dependency tracker.

Class Structure

Heap

Top-level arena allocator. Manages root stage, stage stack, and thread-local default. Provides static methods for scoped staging and dependency registration.

HeapStage

Public inner class. Bump allocator over a pre-allocated Bytes block. Tracks allocated entries and a HeapDependencies instance. Implements Destroyable.

HeapDependencies

Private inner class. Tracks three categories of resources (dependent operations, compiled operations, created memory) for automatic cleanup when the stage is destroyed.

Bytes

Lightweight memory wrapper returned by HeapStage.allocate(). Supports zero-copy delegation, atomic elements, and range views.

Allocation Mechanism

When HeapStage.allocate(count) is called:

  1. Checks end + count <= data.getMemLength()
  2. Creates new Bytes(count, data, end) — a zero-copy view into the backing block at the current bump pointer
  3. Advances the bump pointer: end += count
  4. Adds the new Bytes to the entries list

Because the returned Bytes delegates to the stage's backing block, no new memory is allocated from the provider. This is the key performance advantage.

API Reference

Constructors

Constructor Description
Heap(int size) Creates a heap with root size size and stage size size / 4. Uses default memory provider.
Heap(int rootSize, int stageSize) Creates a heap with explicit root and stage sizes. Uses default memory provider.
Heap(MemoryProvider, int rootSize, int stageSize) Creates a heap using a specific memory provider. When provider is null, stages use new Bytes(size).

Instance Methods

Method Returns Description
getStage() HeapStage Returns the active stage (top of stack, or root if no pushed stages).
allocate(int count) Bytes Allocates from the active stage. Throws IllegalArgumentException if insufficient space.
wrap(Callable<T>) Callable<T> Returns a callable that activates this heap during execution.
use(Runnable) Heap Executes runnable with this heap as default. Restores previous default. Returns this.
use(Supplier<T>) T Executes supplier with this heap as default. Returns the supplier's result.
destroy() void Destroys all stages (pushed + root) and all tracked dependencies. Synchronized.

Static Methods

Method Returns Description
getDefault() Heap Returns the thread-local default heap, or null.
stage(Runnable) void Pushes a new stage, runs the code, pops and destroys the stage. No-op if no default heap.
addOperation(Supplier<T>) Supplier<T> Registers a dependent operation for cleanup. No-op if no default heap.
addCompiled(T extends OperationAdapter) T Registers a compiled operation for cleanup. No-op if no default heap.
addCreatedMemory(T extends MemoryData) T Registers created memory for cleanup. No-op if no default heap.
Key Design Property: All static methods (stage(), addCompiled(), addOperation(), addCreatedMemory()) are no-ops when no default heap is active. This means code can call them unconditionally — they simply pass through when no heap is present.

HeapStage API

Method Returns Description

Usage Patterns

Basic Arena Allocation

// Create a heap for temporary work
Heap heap = new Heap(50_000);

heap.use(() -> {
    // All allocations come from the heap's pre-allocated block
    Bytes workspace1 = Heap.getDefault().allocate(1000);
    Bytes workspace2 = Heap.getDefault().allocate(500);

    // PackedCollection.factory() also uses the heap when active
    PackedCollection temp = PackedCollection.factory().apply(200);
});

heap.destroy();

Scoped Stages for Batch Processing

Heap heap = new Heap(100_000);
heap.use(() -> {
    // Root-level long-lived allocation
    Bytes persistent = Heap.getDefault().allocate(5000);

    for (int i = 0; i < 100; i++) {
        Heap.stage(() -> {
            // Per-iteration temporaries
            PackedCollection temp = PackedCollection.factory().apply(100);
            Runnable op = someComputation.get();
            op.run();
            // temp and compiled op freed here
        });
    }

    // persistent still valid
});
heap.destroy();

Nested Stages

Heap.stage(() -> {
    Bytes outer = Heap.getDefault().allocate(100);

    Heap.stage(() -> {
        Bytes inner = Heap.getDefault().allocate(50);
        // inner freed when this stage exits
    });

    // outer still valid; inner is now invalid
});
// outer freed when this stage exits

Deferred Execution with wrap()

Heap heap = new Heap(10_000);
Callable<PackedCollection> deferred = heap.wrap(() -> {
    return someProducer.get().evaluate();
});

// Later, in a different context:
PackedCollection result = deferred.call();
// heap was active during the call, now restored
Do not hold references past stage exit. Allocations returned by allocate() become invalid when their stage is destroyed. Accessing them after stage exit is undefined behavior.

Framework Integration

When a default heap is active, several framework components automatically integrate with it:

PackedCollection.factory()

Checks Heap.getDefault(). When non-null, returns a factory that allocates from the heap via heap::allocate instead of creating standalone PackedCollection instances. Any code that creates collections via factory() automatically benefits from arena allocation.

// From PackedCollection:
public static IntFunction<PackedCollection> factory() {
    Heap heap = Heap.getDefault();
    return heap == null ? PackedCollection::new : factory(heap::allocate);
}

DefaultComputer.compileRunnable()

Every compiled AcceleratedComputationOperation is registered via Heap.addCompiled(). When the enclosing stage exits, the operation's destroy() releases its ScopeInstructionsManager and native kernel resources.

// From DefaultComputer:
public Runnable compileRunnable(Computation<Void> c) {
    return Heap.addCompiled(
        new AcceleratedComputationOperation<>(getContext(c), c, true));
}

ProcessDetailsFactory

When creating temporary buffers for kernel arguments that need sized destinations, registers them via Heap.addCreatedMemory(). This ensures temporary argument buffers are freed when the stage exits.

// From ProcessDetailsFactory:
MemoryData result = kernelArgEvaluables[i].createDestination(size);
Heap.addCreatedMemory(result);

getDefaultDelegate() Pattern

Several MemoryDataAdapter subclasses override getDefaultDelegate() to return Heap.getDefault(). When init() is called with a heap active, memory is allocated from the heap instead of the global provider.

Dependency Tracking

Each HeapStage tracks three categories of resources via its HeapDependencies instance:

Category Type Registered By Destroy Action
Dependent operations List<Supplier> Heap.addOperation() destroy() if Destroyable
Compiled operations List<OperationAdapter> Heap.addCompiled() OperationAdapter.destroy()
Created memory List<MemoryData> Heap.addCreatedMemory() MemoryData.destroy()

Relationship to Instruction Caching

Heap dependency tracking and the instruction cache in DefaultComputer are independent systems:

  • Heap tracks compiled operation instances for lifecycle management. When a stage is destroyed, the operation wrapper is destroyed.
  • Instruction cache (DefaultComputer.instructionsCache, a FrequencyCache with capacity 500) caches ScopeInstructionsManager instances by computation signature. The same compiled kernel can be reused by future operations with the same signature.

When a stage destroys a compiled operation, the underlying ScopeInstructionsManager may still exist in the instruction cache if its signature has not been evicted. A future compilation with the same signature will reuse the cached kernel, performing only argument substitution via ProcessArgumentMap.putSubstitutions() instead of full recompilation.

Key insight: Heap.stage() does not invalidate the instruction cache. It frees only the operation wrapper and its per-invocation resources. Kernel reuse across stages is handled by the instruction cache, not the heap.

Lifecycle Management

Stage Lifecycle

1
Creation: push() creates a HeapStage with a backing block of size stageSize
2
Allocation: allocate(count) advances the bump pointer and returns a Bytes view
3
Tracking: addCompiled(), addOperation(), addCreatedMemory() register resources with the stage's dependencies
4
Destruction: pop() calls HeapStage.destroy(), which clears entries, resets the pointer, destroys the backing block, and destroys all dependencies

Heap Lifecycle

1
Construction: Allocates root stage with specified size
2
Activation: use(Runnable) or use(Supplier) sets this heap as the thread-local default
3
Operation: Allocations and registrations target the active stage (root or pushed)
4
Stage cycling: stage(Runnable) creates temporary nested stages for scoped cleanup
5
Destruction: destroy() pops all pushed stages and destroys root

Dependency Destroy Order

When HeapDependencies.destroy() is called, resources are freed in this order:

  1. Dependent operations — they may reference compiled ops or memory
  2. Compiled operations — kernel teardown may access argument memory
  3. Created memory — safe to free after all references are released

Each list is set to null after processing to prevent double-destruction and release references for garbage collection.

Thread Safety

Aspect Guarantee
Thread-local default Each thread has its own default heap via ThreadLocal. Different threads cannot interfere.
HeapStage.allocate() Synchronized to guard against re-entrant allocation (e.g., during compilation callbacks).
Heap.destroy() Synchronized to prevent concurrent destruction.
Cross-thread sharing Not supported. A heap instance should only be used by one thread at a time.

Common Pitfalls

Holding References Past Stage Exit

Bytes allocated from a stage become invalid when that stage is destroyed. Do not store references to stage-allocated memory outside the stage's scope.

// WRONG: retained reference is invalid after stage exits
Bytes[] retained = new Bytes[1];
Heap.stage(() -> {
    retained[0] = Heap.getDefault().allocate(100);
});
retained[0].setMem(0, 42.0);  // UNDEFINED BEHAVIOR

Exceeding Stage Capacity

Each stage has a fixed capacity set at construction. Allocating more than the stage can hold throws IllegalArgumentException. Size the heap appropriately for your workload.

Heap heap = new Heap(1000);
heap.use(() -> {
    heap.allocate(800);
    heap.allocate(300);  // IllegalArgumentException: No room remaining
});

Creating Guarded Components

AudioSumProvider and AudioProcessingUtils must be created before activating a heap. They throw RuntimeException if Heap.getDefault() != null.

Forgetting to Destroy

Always call heap.destroy() when finished. The root stage's backing memory is not freed until destroy() is called.
