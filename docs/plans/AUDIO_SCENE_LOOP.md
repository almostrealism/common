# AudioScene Loop Computation: Runtime Optimization

## Context

The real-time AudioScene renderer compiles the entire per-frame DSP pipeline
into a single native C function executed via JNI. With 6 channels and the
full effects pipeline enabled, the generated loop body contains **~411 lines
of actual computation** (plus ~71,000 lines of metadata comments when
`AR_HARDWARE_METADATA=enabled`). The function receives **~300+ arguments**
(post-consolidation), runs 4096–16384 iterations per buffer tick, and must
complete within the buffer's playback duration (~93–372 ms at 44.1 kHz).

Current measurements show runtime is **slightly slower than real-time** — the
focus of this document is identifying concrete optimization opportunities in
the generated code to bring it under budget.

### How to Reproduce

Run `effectsEnabledPerformance` in `AudioSceneBufferConsolidationTest` with
instruction set monitoring and metadata enabled:

```bash
mvn test -pl compose \
  -DargLine="-DAR_HARDWARE_METADATA=enabled -DAR_INSTRUCTION_SET_MONITORING=always" \
  -Dtest=AudioSceneBufferConsolidationTest#effectsEnabledPerformance
```

Generated code is written to `compose/results/jni_instruction_set_*.c`. The
largest file (~11 MB) is the Loop computation. Strip comment lines
(`grep -v '^\s*//'`) to see the ~411 lines of actual code.

## Generated Code Structure

The loop body has this high-level structure, repeating per channel:

```
for (int i = 0; i < bufferSize;) {
    // 1. WaveCellPush — bounds-checked buffer reads (24 total, 2 per channel × 2 stereo × 6 channels)
    // 2. Copy-Zero-Accumulate — CachedStateCell tick pattern
    // 3. Envelope/Automation — complex math expressions per channel
    // 4. TimeSeries append + lookup — delay line write and interpolated read
    // 5. Frame reset if-else chain — multi-branch frame position check
    // 6. Frame index increment
    i = i + 1;
}
```

## Optimization Opportunities

### 1. Eliminate Redundant Intermediary Assignments (Copy-Zero-Accumulate)

**Pattern observed:**
```c
temp[offset] = source[offset];      // copy to temp
source[offset] = 0.0;               // zero source
accumulator[offset] = accumulator[offset] + temp[offset];  // accumulate
```

This three-step pattern appears **~50+ times** in the loop, generated by
`CachedStateCell.tick()` which copies a cached value to output, zeros the
cache, then pushes downstream. The temp variable is immediately consumed
and never used again.

**Opportunity:** For single-consumer cells (which most are in this pipeline),
the copy-to-temp step is unnecessary. The accumulation can be written
directly:

```c
accumulator[offset] = accumulator[offset] + source[offset];
source[offset] = 0.0;
```

This eliminates one memory read, one memory write, and one temporary
variable per cell per frame. With ~50 occurrences × 16384 iterations, this
removes **~1.6 million** unnecessary memory operations per tick.

**Implementation approach:** At the `Scope` or `ExpressionAssignment` level,
detect when a variable is written then immediately read exactly once, and
inline it. Alternatively, `CachedStateCell.tick()` could generate a fused
"accumulate-and-reset" operation when the downstream receptor is known to be
a `SummationCell`. This would require `CachedStateCell` to have awareness of
its receptor type, or a scope-level optimization pass that recognizes the
`assign → reset → add` triple.

### 2. Hoist Loop-Invariant Subexpressions

**Pattern observed:**
```c
// Repeated every frame, but values don't change within a buffer tick
double f_assignment_0 = (pow((- pow(genome[offset + 5], 3.0)) + 1.0, -1.0) + -1.0) * -60.0;
result = ... (frameCounter / 44100.0) + f_assignment_0 ...
```

Many subexpressions depend only on genome parameters and constants — they
do not change across the 4096–16384 iterations of the inner loop. Examples:

- **Genome-derived envelope parameters**: `pow((- pow(genome[x], 3.0)) + 1.0, -1.0)`
  appears dozens of times, always reading the same genome values
- **Sample rate division**: `frameCounter / 44100.0` — the `44100.0` constant
  appears **460 times** in the loop body across various expressions
- **Automation LFO parameters**: Expressions like
  `((genome[offset + 42] * 2.0) + -1.0) * 0.5` are recomputed every frame
  but depend only on genome values

**Opportunity:** Move computations that depend only on kernel arguments (not
on the loop index or any value mutated within the loop) outside the loop.
This is classic loop-invariant code motion (LICM).

**Implementation approach:** During scope simplification, classify each
`ExpressionAssignment` as loop-variant or loop-invariant based on whether
its inputs include the loop index variable or any variable assigned inside
the loop body. Loop-invariant assignments can be hoisted into the parent
scope (before the `Repeated` block). The C compiler may do some of this, but
the deeply nested expression trees and pointer-through-offset memory access
patterns make it difficult for the C optimizer to prove invariance.

### 3. Replace Linear if-else Chain with Indexed Lookup for Frame Reset

**Pattern observed (end of loop body):**
```c
if ((reset[offset + 1] > 0.0) & (((- reset[offset]) + frame[offset + 1]) == 0)) {
    frame[offset] = 0.0;
} else if ((reset[offset + 1] > 0.0) & (((- reset[offset + 1]) + frame[offset + 1]) == 0)) {
    frame[offset] = 0.0;
} else if ((reset[offset + 1] > 0.0) & (((- reset[offset + 2]) + frame[offset + 1]) == 0)) {
    frame[offset] = 0.0;
}
// ... continues for 30+ branches, all doing the same thing
```

This is a linear search through a list of frame-reset triggers. Each branch
checks whether the current frame matches a specific trigger value. All
branches execute the same action (`frame = 0.0`). The chain has **30+
entries**, meaning on average ~15 comparisons per frame.

**Opportunity:** Replace the if-else chain with a single loop:

```c
for (int j = 0; j < resetCount; j++) {
    if ((reset[offset + 1] > 0.0) && (frame[offset + 1] == reset[offset + j])) {
        frame[offset] = 0.0;
        break;
    }
}
```

Or, since the reset values are known at compilation time and don't change,
pre-sort them and use a binary search, or use a bitset/hash lookup.

**Implementation approach:** This pattern is generated by the `WaveCell`
frame reset logic, which is itself generated by iterating over pattern
elements. The code generator currently unrolls each element into its own
`else if` branch. A more efficient approach would be to recognize this
pattern during scope generation (or as a post-processing pass on the
`Scope` tree) and consolidate the identical-action branches into a loop
over an array argument.

### 4. Reduce Argument Count via Further Buffer Consolidation

**Current state:** After render buffer and filter buffer consolidation, there
are still **~300+ arguments** passed to the kernel. Each argument requires:
- JNI array element extraction (`GetLongArrayElements`)
- Offset extraction from `offsetArr`
- Size extraction from `sizeArr`

This JNI marshalling overhead is paid **once per tick** (not per iteration),
but with 300+ arguments it adds up. More importantly, large argument counts
stress the C compiler's register allocator and can prevent optimization of
the inner loop.

**Remaining consolidation targets (from metadata analysis):**

| Category | Count | Consolidation Potential |
|----------|-------|------------------------|
| WaveCell state (cursor, bounds, gain) | ~24 × 6 fields | High — pack per-cell state into one contiguous array |
| Automation TimeSeries buffers | ~12 | Medium — share one delay buffer pool |
| Frame reset trigger arrays | ~12 | High — one consolidated trigger array |
| Genome parameter collections | ~6 (post-consolidation) | Already done |
| Render output buffers | 1 (consolidated) | Already done |
| Filter destination buffers | 1 (consolidated) | Already done |

The **WaveCell state collections** are the largest remaining opportunity.
Each WaveCell has a ~20-element state array (cursor position, buffer start,
buffer end, gain, etc.) and there are 24 WaveCells. If these were packed into
a single contiguous allocation (like the render buffers were), the argument
deduplication would collapse all 24 into one.

**Implementation approach:** Follow the same pattern as
`AudioScene.consolidateRenderBuffers()` — pre-allocate a single contiguous
`PackedCollection` for all WaveCell state, assign delegate ranges to each
cell. The scope's argument deduplication resolves delegates to the shared
root, collapsing arguments.

## Implementation Status (reviewed 2026-02-27)

> **IMPORTANT FOR IMPLEMENTOR:** A previous attempt at these optimizations
> resulted in **zero measurable performance gain**. The root cause was that the
> only high-impact optimization (#2 LICM) was implemented but then **disabled
> by default** due to correctness bugs, and the remaining work was either
> skipped or addressed only cosmetically. Read the assessment below carefully
> before starting work. Each optimization requires studying how the AR
> expression system and code generation pipeline actually work — superficial
> changes to Java source that don't change the generated C output have no
> performance effect.

### #1 — Copy-Zero-Accumulate: NOT ATTEMPTED

`CachedStateCell.tick()` (lines 200–208 in
`graph/src/main/java/org/almostrealism/graph/CachedStateCell.java`) is
completely unchanged. It still generates the three-step
`assign(outValue, cachedValue)` → `reset(cachedValue)` → `push(null)` pattern.
No scope-level optimization pass to detect and fuse the resulting
`assign → zero → add` triple was introduced either.

**What needs to happen:** Either modify `CachedStateCell.tick()` to generate a
fused operation when the downstream receptor is a `SummationCell`, or add a
post-processing pass at the `Scope` level that recognizes the triple and
inlines the temporary. The key is that the **generated C code** must change —
the three memory operations per cell per iteration must become two. Verify by
running the reproduction test (see "How to Reproduce" above), stripping
comments from the generated `.c` file, and confirming the pattern is gone.

### #2 — Loop-Invariant Code Motion: IMPLEMENTED BUT DISABLED

~200 lines were added to `Repeated.java` implementing LICM in `simplify()`.
The feature is gated behind `AR_LOOP_INVARIANT_HOISTING` (defaults to `false`).

**Why it was disabled:** The initial implementation only checked
`containsReference(Variable)` for the loop index, but AR's expression system
represents loop indices via the `Index` interface, not `Variable` references.
Expressions referencing nested loop indices (e.g., inside `timeSeriesValueAt`
helper functions) were incorrectly classified as invariant and hoisted,
producing C code with undeclared variables. A fix was added using
`containsIndex(Index)` checks, but the author was not confident it was
complete, so the feature remains disabled.

**Known remaining issues with the current implementation:**

1. **`getDependencies()` completeness** — The invariance check relies on
   `expr.getDependencies()` returning a complete transitive closure of all
   variable references. This is not guaranteed across all `Expression`
   subclasses. Some subclasses may represent dependencies through mechanisms
   other than child expressions.

2. **Single-level hoisting** — `hoistLoopInvariantStatements` only iterates
   over `scope.getChildren()` and their `getStatements()`. It does not recurse
   into grandchildren. Invariant declarations nested two or more levels deep
   are not hoisted.

3. **Destination variable extraction** — `collectLoopAssignedVariables` calls
   `assignment.getDestination().getDependencies()` to extract the assigned
   variable name. For complex destination expressions (e.g., array elements),
   this indirect approach may not correctly identify the target variable.

4. **No side-effect analysis** — Only declarations are hoisted, which is
   conservative, but there is no verification that the hoisted expression is
   free of side effects.

**What needs to happen:** Study the `Expression` type hierarchy — specifically
how `Index`, `Variable`, `InstanceReference`, `containsReference()`,
`containsIndex()`, and `getDependencies()` interact. Build a set of unit tests
with nested loops, `timeSeriesValueAt`-style helper functions, and
genome-parameter-only expressions. Verify that the invariance analysis
correctly classifies each case. Only then enable the feature by default. The
generated C output must show genome-derived `pow()` and `sin()` expressions
moved above the `for` loop. There are hundreds of these — they should be
visible in the non-comment lines.

### #3 — Frame Reset if-else Chain: COSMETIC REFACTOR ONLY

The previous attempt removed a redundant `if (len == 1)` branch in
`TimeCellReset.java` that was identical to the `len > 1` loop body. This is a
valid Java source cleanup, but **the generated C code is completely
unchanged** — it still produces a 30+ branch if-else chain with one branch per
reset slot.

The code includes a comment acknowledging this:
```java
// Note: We considered using a loop for more compact generated code, but the
// HybridScope's direct code generation doesn't automatically include
// variable declarations, making loop-based access problematic.
```

**What needs to happen:** The goal is to change the **generated C output**
from a 30+ branch if-else chain to a compact loop. The comment about
HybridScope limitations suggests the implementor hit a real obstacle with how
`HybridScope.code()` emits raw string fragments without proper variable
tracking. Possible approaches:

- Generate a proper `Repeated` (loop) scope inside `TimeCellReset` instead of
  using raw string emission via `HybridScope.code()`. This would require
  restructuring `prepareScope()` to build a scope tree rather than emitting
  string fragments.
- Alternatively, pass `resetCount` as an argument and emit a raw C loop
  (`for (int j = 0; j < resetCount; j++)`) with the reset array indexed by
  `j`. The variable declaration issue only matters if using AR's expression
  tracking — a raw string loop variable sidesteps it.

Verify by checking the generated `.c` file: the 30+ identical if-else
branches should be replaced by a single loop construct.

### #4 — Further Buffer Consolidation: NOT ATTEMPTED

No changes were made to WaveCell state packing or argument consolidation.

**What needs to happen:** Follow the pattern established by
`AudioScene.consolidateRenderBuffers()` — pre-allocate a single contiguous
`PackedCollection` for all WaveCell state arrays, assign delegate ranges to
each cell. Study the existing consolidation code before implementing. The
scope's argument deduplication (which resolves delegates to shared roots)
should collapse the ~24 × 6 WaveCell state arguments into one. Verify by
checking the argument count in the generated C function signature.

## Prioritization

| # | Opportunity | Estimated Impact | Complexity | Status |
|---|-------------|-----------------|------------|--------|
| 2 | Hoist loop-invariant expressions | High | Medium | Disabled — needs invariance analysis fixes |
| 1 | Eliminate copy-zero-accumulate intermediaries | Medium | Low–Medium | Not started |
| 3 | Replace if-else chain with loop | Low–Medium | Low | Cosmetic only — generated code unchanged |
| 4 | Further buffer consolidation | Medium (indirect) | Medium | Not started |

**Recommendation:** Focus on **#2 (LICM)** first since ~200 lines of
implementation already exist — the task is to fix the invariance analysis and
enable it, not to rewrite from scratch. Study the `Expression` / `Index` /
`Variable` type hierarchy thoroughly before touching the code.

Then tackle **#1 (copy-zero-accumulate)** as it is the most mechanical
optimization with the clearest before/after verification.

**#3 and #4** are lower priority but achievable once the first two are
delivering measurable gains.

### Verification Protocol

For every optimization, the acceptance criterion is a change in the
**generated C code**, not just the Java source. Use this workflow:

1. Run the reproduction test (see "How to Reproduce" above)
2. Find the largest generated `.c` file in `compose/results/`
3. Strip comments: `grep -v '^\s*//' <file> > stripped.c`
4. Compare the stripped output against the baseline (~411 non-comment lines)
5. Confirm the targeted pattern is reduced or eliminated

If the generated C code looks the same before and after your change, **the
optimization is not working** regardless of what the Java source looks like.
