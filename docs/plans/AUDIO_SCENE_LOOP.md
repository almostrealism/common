## Overview

The real-time AudioScene renderer compiles a per-frame DSP pipeline into a
single native C function executed via JNI. With 6 channels and the full
effects pipeline enabled, the inner loop runs 4096 iterations per buffer
tick and must complete within ~93 ms (44.1 kHz playback). Currently it runs
**slightly slower than real-time**.

The dominant cost is **126 `pow()` calls per iteration** in the inner loop.
LICM (loop-invariant code motion) has already extracted 133 sub-expressions,
but many `pow()` calls remain because they are embedded in expressions that
also depend on the loop variable. Reducing the per-iteration `pow()` count
is the primary path to achieving real-time performance.

## Goals

### Goal 1: Reduce inner-loop `pow()` count below 30

**Status: INCOMPLETE — currently 126, target < 30**

Each of the 12 envelope accumulate expressions contains ~10 `pow()` calls.
Some of these are genome-only sub-expressions that should be loop-invariant
but are nested inside larger expressions that depend on the loop variable
`_32879_i`. Example from generated code:

```c
// The pow(genome[offset + 1], 3.0) sub-expression is invariant,
// but the enclosing expression depends on the frame counter
fmin(fmax((...frame_counter...) * pow((- pow(genome[offset + 1], 3.0)) + 1.0, -1.0), 0.0), 1.0)
```

Phase 4 already extracts SOME genome-only `pow()` sub-expressions (133
extractions), but 126 `pow()` calls remain in the inner loop.

**Required work:**

1. **Classify the remaining 126 `pow()` calls.** Read the generated C file
   and determine for each `pow()` call whether it depends on the loop
   variable (`_32879_i`) or only on kernel arguments (`argArr[N]`,
   `genome[offset + N]`). Document the counts in the journal.

2. **For invariant `pow()` calls that Phase 4 misses:** Investigate why
   `isLoopInvariant()` in `Repeated.java` doesn't identify them. The issue
   is likely in how the expression tree represents array accesses to kernel
   arguments. Fix the invariance analysis to extract them.

3. **For truly loop-variant `pow()` calls:** Implement algebraic strength
   reduction. `pow(x, 3.0)` should become `x*x*x`. `pow(x, -1.0)` should
   become `1.0/x`. `pow(x, 2.0)` should become `x*x`. These avoid the
   expensive `pow()` library call entirely. Look at `LanguageOperations` or
   the expression simplification pipeline in `Scope.simplify()` for where
   to implement this. These are standard compiler optimizations.

**Acceptance criteria:**

- [ ] Generated C file from `effectsEnabledPerformance` has < 30 `pow()`
  calls inside the inner `for` loop
- [ ] All existing tests pass (`code`, `hardware`, `audio`, `compose`)
- [ ] Analysis of the 126 `pow()` calls documented in decision journal

## Verification Protocol

**Every goal requires verification against the generated C code, not just
the Java source.** Previous optimization attempts produced Java changes that
appeared correct but did not change the generated output.

### How to run the verification test

```
mcp__ar-test-runner__start_test_run
  module: "compose"
  test_methods: [{"class": "AudioSceneBufferConsolidationTest",
                  "method": "effectsEnabledPerformance"}]
  jvm_args: ["-DAR_INSTRUCTION_SET_MONITORING=always"]
  timeout_minutes: 15
```

Generated `.c` files appear in `compose/results/<run_id>/`. The largest
file is the loop kernel.

### What to check

1. Strip comments: `grep -v '^\s*//' <file> > stripped.c`
2. Find the inner loop: `for (int _*_i = 0; _*_i < 4096;)`
3. Count `pow()` between that line and the matching closing `}`
4. Verify all `f_assignment` declarations appear BEFORE the inner loop
5. Count `f_licm` declarations (should be >= 133, ideally more)

**If the generated C code is unchanged after your modifications, the
optimization is not working.** Do not claim completion based on unit
tests alone — you must inspect the generated code.

The test has synthetic sample fallback and works without real audio files.
If it fails, document the exact error in the decision journal.
