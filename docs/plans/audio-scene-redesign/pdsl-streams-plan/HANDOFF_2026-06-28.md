## 4. The genuinely open question (this is the actual work)

**Why does each fresh AudioScene re-pay ~143 s of setup in the same JVM, given the instruction
cache is JVM-wide and signatures are structural?** This is unresolved. The honest candidate set:

1. The AudioScene's real kernels *do* sign structurally (like the `add` case) and the cache *is*
   reused, but the ~143 s is **not compilation** — it is per-scene work (e.g. a2 audio rendering
   during the render-ahead ring fill, scene construction, etc.) that I mislabeled "compile." 
2. The kernels are genuinely recompiled because (a) the AudioScene builds kernels whose signatures
   contain a non-structural component the simple `add` case doesn't exercise, or (b) >500 distinct
   signatures cause `FrequencyCache` eviction (capacity 500, eviction listener **destroys** the
   native kernel), or (c) the compile path isn't consulting the cache for these kernels.

These have **opposite** strategic implications (fix caching vs. fix compile-perf vs. it's-not-compile),
so resolving it is the prerequisite to any direction.

### How to resolve it (measure, don't infer)
- **Instrument the cache directly.** Add hit/miss (or native-compile) counters to
  `DefaultComputer.getScopeInstructionsManager` / `instructionsCache` (or the
  `ScopeInstructionsManager` compile path) and read them across scene 0 vs scene 1 in
  `KernelCacheReuseAcrossScenesTest`. If scene 1 is ~all hits, it is NOT a caching problem and the
  ~143 s is something else — find what. If scene 1 is ~all misses, dump the missed signatures and
  diff scene0 vs scene1 to see exactly what differs.
- **Decompose the 143 s setup** of one scene (compile time vs render time vs other) — e.g. with an
  `OperationProfile` over `setup.run()`, or `get_timing_breakdown` (ar-profile-analyzer) splitting
  compile vs run, the same way the per-buffer forward was decomposed.
- Only after one of those returns a number should anyone write "the priority is X."

## 5. Tooling note (real, diagnosed)

`ar-jmx` JFR/allocation profiling of a forked test JVM is **blocked** for large modules:
`tools/mcp/test-runner/server.py` `_discover_forked_pid` (~L773) polls `jps` for only `range(30)`
(~30 s), but `studio/compose` (105 test files) takes ~60 s to test-compile before surefire forks
the JVM, so discovery times out → `forked_pid_discovery_failed` → `attach_to_run` fails. Fix:
poll while the maven process is alive (or raise the bound). Editing it does **not** reload the
running MCP server in-session. This is why the Java/native split of the per-dispatch cost (§3) was
never measured.

## 8. RESOLUTION (2026-06-28, run `68bfcc17`) — §4 answered: there is NO caching problem

§4 was resolved by doing what it asked: instrumenting the cache directly. `DefaultComputer` got
hit/miss/eviction counters (`instructionCacheLookups` / `instructionCacheMisses` /
`instructionCacheEvictions` + `resetCacheCounters()`), read per scene in
`KernelCacheReuseAcrossScenesTest` (2 scenes, seed 58, 8192-frame buffers, 180 ticks, full efx+reverb).

| scene | setupMs | renderLoopMs | firstTickMs | instrMisses | instrHits | instrEvictions |
|---|---|---|---|---|---|---|
| 0 (cold) | 130255 | 69570 | 135.5 | 30 | 29 | 0 |
| 1 (warm) | 131472 | 70060 | 60.6 | **0** | 54 | 0 |

**The instruction cache reuses kernels across scenes flawlessly.** Scene 1 recompiled zero kernels
(`instrMisses=0`; all 54 lookups hit), `instrEvictions=0`, and only ~30 distinct native kernels exist
total (far under the 500 capacity). This **rules out candidate 2 entirely** (2a non-structural
signature, 2b eviction thrash, 2c cache-not-consulted) and **confirms candidate 1**: the ~130 s setup
and ~70 s render per scene are *genuine per-scene rendering compute, not compilation*. The identical
per-scene wall times the original session read as a caching failure are identical because the *work* is
identical and real. (Corroborated by `firstTickMs`: the mixdown's first tick is trivial in both scenes,
so the mixdown kernel is reused, not recompiled.)

**Red herring:** `rendererCompiles=7` in both scenes is a per-instance Java map
(`BatchedPatternLayerRenderer.rendererCache`) being rebuilt — not native recompilation. Judge native
reuse by `DefaultComputer.instructionCacheMisses`, never by `rendererCompiles`.

**The real bottleneck is a2 pattern/note rendering** (setup is dominated by `patternSetup` + the
`renderStream` prefill; ~397 note-audio renders per scene; the render loop waits on
`renderStream.awaitSlot`). `SOURCE_BUCKET` is now 16384 (was 1024), so a2 kernel-shape thrash is already
mitigated (only 7 distinct shapes). Next performance work belongs on a2 throughput, not caching.

## 9. CORRECTION (2026-07-04) — §8's conclusion was also wrong; the "red herring" was the cost

§8's *receipts* stand (`instrMisses=0` on scene 1 — the native instruction cache reuses
flawlessly; there is no caching problem). Its *conclusion* — that the ~130 s per-scene setup is
"genuine per-scene rendering compute" and the next work is "a2 throughput" — is **disproven by
instrumentation** (see `../SETUP_FRONT_LOADING_HANDOFF.md`, RESOLVED section):

- Stage-level timing of the setup `OperationList` (`PdslSetupBreakdownTest`) plus `jstack`
  sampling attributed ~129 s of the ~130 s to **seven first-`evaluate()` calls of the batched
  dispatch kernels** — Java-side `Evaluable` instantiation, not rendering and not native compile.
- The mechanism: `CollectionSumComputation` unconditionally enables the `replaceLoop`
  gather-collapse probe; `AggregatedProducerComputation.prepareScope` →
  `TraversableExpression.uniqueNonZeroOffset` then builds an `ExpressionMatrix` over
  `[windowWidth × bucketN]`, substituting each index into the deep fused chain. The batched
  chains end in a scatter-add over **overlapping** notes, so the probe can never succeed — it
  burned **14–29 s per kernel shape** and returned null every time.
- **The `rendererCompiles=7` that §8 dismissed as a red herring was in fact the cost**: those 7
  per-instance builds were exactly the 7 probe payments. The lesson inverts cleanly: judge
  *native* reuse by `instructionCacheMisses` (correct), but do not assume a per-instance Java
  rebuild is therefore cheap.
- Fix: `BatchedPatternRenderer.sumNoteAxis` disables `replaceLoop` on the note-axis sums.
  Measured after (pinned scene, seed 58, 8192): per-scene setup **128.6 s → ~2.3 s**
  (`KernelCacheReuseAcrossScenesTest`: 2306/2136 ms for scenes 0/1), honest
  `generateRealtimeX` **0.63 → 1.46** (`GenerateAudioFileTest`, 2026-07-03), rendered peak
  bit-identical. The same probe firing on lazily-compiled shapes mid-stream was the ~29–33 s
  "compile spike" that motivated the whole-arrangement pre-warm; with the probe gone the
  pre-warm was removed (`preWarmMaxSeconds` default 0) as front-loaded rendering.
