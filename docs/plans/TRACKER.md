# Tracker: Lightweight Internal Ticket System

## Goal

Replace Jira for internal work organization with a minimal, self-hosted ticket-tracking
service. The system is intentionally simple — it captures tasks, groups them by project
and release, and ties them back to the existing FlowTree workstream model. No assignees,
no sprints, no comments threads, no time-tracking. Just enough structure to organize work
before it becomes code.

The deployment mirrors the existing `ar-memory` service: a Python REST API running in
Docker, persisting data to `/Users/Shared/flowtree/tracker/`, with `ar-manager` as the
primary external interface.

## 1. Data Model

### 1.1 Database Choice

**SQLite** — the same backend used by `ar-memory`. Justifications:
- Single-file database makes backup and disaster recovery trivial
- No separate database process to manage
- The workload is read-heavy with low concurrency (MCP tools, occasional imports)
- Consistent with the established service pattern in this project

Postgres would be warranted if we expect concurrent writes from multiple services, need
row-level locking under load, or plan to add full-text search beyond what SQLite FTS5
provides. None of those apply here.

### 1.2 Schema

```sql
CREATE TABLE projects (
    id          TEXT PRIMARY KEY,   -- UUID v4 (generated by service)
    name        TEXT NOT NULL,
    created_at  TEXT NOT NULL       -- ISO 8601 UTC
);

CREATE TABLE releases (
    id          TEXT PRIMARY KEY,   -- UUID v4
    name        TEXT NOT NULL,
    project_id  TEXT REFERENCES projects(id) ON DELETE SET NULL,
    created_at  TEXT NOT NULL       -- ISO 8601 UTC
);

CREATE TABLE tasks (
    id              TEXT PRIMARY KEY,   -- UUID v4
    title           TEXT NOT NULL,
    description     TEXT,               -- Markdown; may be NULL
    status          TEXT NOT NULL DEFAULT 'open', -- 'open' | 'closed'
    project_id      TEXT REFERENCES projects(id) ON DELETE SET NULL,
    release_id      TEXT REFERENCES releases(id) ON DELETE SET NULL,
    workstream_id   TEXT,               -- FlowTree workstream ID; no FK (external)
    created_at      TEXT NOT NULL,      -- ISO 8601 UTC
    updated_at      TEXT NOT NULL       -- ISO 8601 UTC
);

-- Lookup indexes
CREATE INDEX idx_tasks_project    ON tasks(project_id);
CREATE INDEX idx_tasks_release    ON tasks(release_id);
CREATE INDEX idx_tasks_workstream ON tasks(workstream_id);
CREATE INDEX idx_tasks_status     ON tasks(status);

-- Full-text search via SQLite FTS5
-- content= keeps FTS in sync with the tasks table automatically
CREATE VIRTUAL TABLE tasks_fts USING fts5(
    title,
    description,
    content=tasks,
    content_rowid=rowid
);

-- Triggers to keep FTS index in sync with tasks table
CREATE TRIGGER tasks_ai AFTER INSERT ON tasks BEGIN
    INSERT INTO tasks_fts(rowid, title, description)
    VALUES (new.rowid, new.title, new.description);
END;

CREATE TRIGGER tasks_ad AFTER DELETE ON tasks BEGIN
    INSERT INTO tasks_fts(tasks_fts, rowid, title, description)
    VALUES ('delete', old.rowid, old.title, old.description);
END;

CREATE TRIGGER tasks_au AFTER UPDATE ON tasks BEGIN
    INSERT INTO tasks_fts(tasks_fts, rowid, title, description)
    VALUES ('delete', old.rowid, old.title, old.description);
    INSERT INTO tasks_fts(rowid, title, description)
    VALUES (new.rowid, new.title, new.description);
END;
```

**Notes on design decisions:**

- `project_id` on tasks: **recommended as required** (see Open Questions §6.2). A task
  with no project is noise. The service should accept it for migration flexibility but
  the MCP layer should guide callers to always supply it.
- `workstream_id` is a string, not a FK, because workstream IDs live in the FlowTree
  controller — the tracker has no knowledge of that data store.
- `status` is currently `'open' | 'closed'`. Additional states (e.g., `'in-progress'`,
  `'blocked'`) can be added later without a schema migration by changing the validation
  layer. See Open Questions §6.1.
- `releases.project_id` is optional: a release can span projects, or a project can have
  no releases yet.

### 1.3 Migration

Schema migrations use a simple `schema_version` table and sequential numbered SQL files,
identical to the pattern used by the FlowTree controller's database layer. No third-party
migration tool is needed.

```sql
CREATE TABLE schema_version (version INTEGER NOT NULL);
INSERT INTO schema_version VALUES (1);
```

Each new schema file (e.g., `002_add_priority.sql`) reads the current version, applies
the migration, and increments it.

## 2. API Service

### 2.1 Language and Framework

**Python + Starlette** — exactly matching the `ar-memory` service (`tools/mcp/memory/`).
This lets future agents copy patterns verbatim: same HTTP middleware, same bearer-token
auth check, same health endpoint structure, same Dockerfile base image.

Dependencies (to match `tools/mcp/memory/requirements.txt` pattern):
```
starlette
uvicorn[standard]
```

No ORM. Raw `sqlite3` module with parameterized queries, matching the memory service's
`store.py` style.

### 2.2 Authentication

Same bearer-token scheme as `ar-memory`: the service reads `AR_TRACKER_AUTH_TOKEN` from
the environment. If unset, the API is open (dev mode). The auth check is a Starlette
middleware function that short-circuits non-health requests without a valid
`Authorization: Bearer <token>` header and returns HTTP 401.

```python
async def _check_auth(request: Request) -> Optional[JSONResponse]:
    if not auth_token:
        return None  # Open in dev mode
    header = request.headers.get("authorization", "")
    if header.startswith("Bearer ") and header[7:].strip() == auth_token:
        return None
    return JSONResponse(
        {"ok": False, "error": "Unauthorized: valid Bearer token required"},
        status_code=401,
        headers={"WWW-Authenticate": 'Bearer realm="ar-tracker"'},
    )
```

The `ar-manager` service stores `AR_TRACKER_AUTH_TOKEN` as an environment variable and
passes it in every outbound request to the tracker.

### 2.3 API Layout

All routes are versioned under `/v1/`. The health endpoint is unversioned (consistent
with `ar-memory`'s `/api/health` pattern).

```
GET  /api/health

# Projects
GET    /v1/projects                     List all projects
POST   /v1/projects                     Create a project
GET    /v1/projects/{id}                Get a project
PUT    /v1/projects/{id}                Update a project
DELETE /v1/projects/{id}                Delete a project

# Releases
GET    /v1/releases                     List all releases (optional ?project_id=)
POST   /v1/releases                     Create a release
GET    /v1/releases/{id}                Get a release
PUT    /v1/releases/{id}                Update a release
DELETE /v1/releases/{id}                Delete a release

# Tasks
GET    /v1/tasks                        List tasks (filterable, paginated)
POST   /v1/tasks                        Create a task
GET    /v1/tasks/{id}                   Get a task
PUT    /v1/tasks/{id}                   Update a task
DELETE /v1/tasks/{id}                   Delete a task

GET    /v1/projects/{id}/tasks          List tasks for a project (pagination)
GET    /v1/releases/{id}/tasks          List tasks for a release (pagination)
GET    /v1/workstreams/{id}/tasks       List tasks for a workstream

# Search
GET    /v1/search/tasks?q=...           Full-text search over title + description

# Bulk import (migration)
POST   /v1/import                       Bulk import projects, releases, tasks
```

### 2.4 Request and Response Shapes

**Create task** (`POST /v1/tasks`):
```json
{
  "title": "Add OAuth support",
  "description": "## Background\n\nWe need...",
  "project_id": "uuid",
  "release_id": "uuid",
  "workstream_id": "ws-abc123",
  "status": "open"
}
```

**Task response**:
```json
{
  "ok": true,
  "task": {
    "id": "uuid",
    "title": "Add OAuth support",
    "description": "## Background\n\nWe need...",
    "status": "open",
    "project_id": "uuid",
    "release_id": "uuid",
    "workstream_id": "ws-abc123",
    "created_at": "2026-05-02T12:00:00Z",
    "updated_at": "2026-05-02T12:00:00Z"
  }
}
```

**List tasks** (`GET /v1/tasks`):
Query params:
- `project_id` — filter by project
- `release_id` — filter by release
- `workstream_id` — filter by workstream
- `status` — filter by status (`open`, `closed`)
- `limit` — page size (default 50, max 200)
- `offset` — for pagination
- `sort` — `created_at` (default) or `updated_at`
- `order` — `desc` (default) or `asc`

Response:
```json
{
  "ok": true,
  "tasks": [...],
  "total": 142,
  "limit": 50,
  "offset": 0
}
```

**Search** (`GET /v1/search/tasks?q=oauth&limit=20&offset=0`):
Uses SQLite FTS5 `MATCH` query. Returns same shape as list endpoint plus a `query` field.

**Bulk import** (`POST /v1/import`):
```json
{
  "projects": [{"id": "uuid", "name": "Rings"}],
  "releases": [{"id": "uuid", "name": "Rings 0.38", "project_id": "uuid"}],
  "tasks": [{"id": "uuid", "title": "...", "project_id": "uuid", ...}]
}
```
The import is idempotent: existing IDs are upserted (UPDATE OR REPLACE). Returns a
summary of created/updated counts per entity type.

## 4. ar-manager MCP Integration

### 4.1 HTTP Helper Functions

Add three new private helpers to `tools/mcp/manager/server.py`, modeled exactly on
`_controller_get` and `_controller_post`:

```python
TRACKER_URL = os.environ.get("AR_TRACKER_URL", "http://ar-tracker:8030")
_TRACKER_AUTH_TOKEN = os.environ.get("AR_TRACKER_AUTH_TOKEN", "")

def _tracker_headers() -> dict:
    h = {"Accept": "application/json"}
    if _TRACKER_AUTH_TOKEN:
        h["Authorization"] = f"Bearer {_TRACKER_AUTH_TOKEN}"
    return h

def _tracker_get(path: str, timeout: int = 10) -> dict:
    url = TRACKER_URL.rstrip("/") + path
    req = Request(url, headers=_tracker_headers())
    print(f"ar-manager: TRACKER GET {url}", file=sys.stderr)
    try:
        with urlopen(req, timeout=timeout) as resp:
            body = resp.read().decode("utf-8")
            return json.loads(body) if body else {}
    except HTTPError as e:
        body = e.read().decode("utf-8", errors="replace")
        try:
            return json.loads(body)
        except json.JSONDecodeError:
            return {"ok": False, "error": f"Tracker returned HTTP {e.code}"}
    except URLError as e:
        return {"ok": False, "error": f"Tracker unreachable: {e.reason}"}
    except Exception as e:
        return {"ok": False, "error": "Internal error contacting tracker"}

def _tracker_post(path: str, payload: dict, timeout: int = 15) -> dict:
    url = TRACKER_URL.rstrip("/") + path
    data = json.dumps(payload, ensure_ascii=False).encode("utf-8")
    h = _tracker_headers()
    h["Content-Type"] = "application/json; charset=utf-8"
    req = Request(url, data=data, headers=h)
    print(f"ar-manager: TRACKER POST {url}", file=sys.stderr)
    try:
        with urlopen(req, timeout=timeout) as resp:
            body = resp.read().decode("utf-8")
            return json.loads(body) if body else {"ok": True}
    except HTTPError as e:
        body = e.read().decode("utf-8", errors="replace")
        try:
            return json.loads(body)
        except json.JSONDecodeError:
            return {"ok": False, "error": f"Tracker returned HTTP {e.code}"}
    except URLError as e:
        return {"ok": False, "error": f"Tracker unreachable: {e.reason}"}
    except Exception as e:
        return {"ok": False, "error": "Internal error contacting tracker"}

def _tracker_put(path: str, payload: dict, timeout: int = 15) -> dict:
    # Same as _tracker_post but with PUT method
    url = TRACKER_URL.rstrip("/") + path
    data = json.dumps(payload, ensure_ascii=False).encode("utf-8")
    h = _tracker_headers()
    h["Content-Type"] = "application/json; charset=utf-8"
    req = Request(url, data=data, headers=h, method="PUT")
    print(f"ar-manager: TRACKER PUT {url}", file=sys.stderr)
    try:
        with urlopen(req, timeout=timeout) as resp:
            body = resp.read().decode("utf-8")
            return json.loads(body) if body else {"ok": True}
    except HTTPError as e:
        body = e.read().decode("utf-8", errors="replace")
        try:
            return json.loads(body)
        except json.JSONDecodeError:
            return {"ok": False, "error": f"Tracker returned HTTP {e.code}"}
    except URLError as e:
        return {"ok": False, "error": f"Tracker unreachable: {e.reason}"}
    except Exception as e:
        return {"ok": False, "error": "Internal error contacting tracker"}

def _tracker_delete(path: str, timeout: int = 10) -> dict:
    url = TRACKER_URL.rstrip("/") + path
    req = Request(url, headers=_tracker_headers(), method="DELETE")
    print(f"ar-manager: TRACKER DELETE {url}", file=sys.stderr)
    try:
        with urlopen(req, timeout=timeout) as resp:
            body = resp.read().decode("utf-8")
            return json.loads(body) if body else {"ok": True}
    except HTTPError as e:
        body = e.read().decode("utf-8", errors="replace")
        try:
            return json.loads(body)
        except json.JSONDecodeError:
            return {"ok": False, "error": f"Tracker returned HTTP {e.code}"}
    except URLError as e:
        return {"ok": False, "error": f"Tracker unreachable: {e.reason}"}
    except Exception as e:
        return {"ok": False, "error": "Internal error contacting tracker"}
```

### 4.2 Workspace Scoping

The tracker integrates with the existing workspace-scoping machinery:

- **Projects and releases** are globally visible. No workspace scope enforcement.
- **Tasks with a `workstream_id`**: When creating or updating a task with a
  `workstream_id`, call `_require_workstream_in_scope(workstream_id)`. This ensures
  workstream-scoped tokens can only create/modify tasks attached to their own workstream.
- **Reading tasks by workstream** (`tracker_list_tasks` with `workstream_id` filter):
  call `_require_workstream_in_scope(workstream_id)` before the GET request.
- **Updating or deleting a task**: fetch the task first, extract its `workstream_id`,
  then call `_require_workstream_in_scope` before applying the mutation.
- **Listing all tasks** (no workstream filter): allowed for all scopes; the result set
  is naturally limited to what the caller requests.

### 4.3 MCP Tool Implementations

All 14 tools follow the exact pattern documented in `tools/mcp/CLAUDE.md`:
`@mcp.tool()` decorator, all parameters in the signature with type hints and defaults,
complete docstring with `Args:` and `Returns:` sections, `_require_scope`, `_audit`.

#### Project Tools

```python
@mcp.tool()
def tracker_list_projects() -> dict:
    """List all tracker projects.

    Returns all projects with their IDs and names. Projects are globally
    visible — no workspace scope restriction.

    Returns:
        Dictionary with ok=True and a list of projects.
    """
    _require_scope("read")
    _audit("tracker_list_projects")
    return _tracker_get("/v1/projects")


@mcp.tool()
def tracker_create_project(name: str) -> dict:
    """Create a new tracker project.

    Args:

        Dictionary with ok=True and the created project record.

        Dictionary with ok=True and the updated project record.

@mcp.tool()

    Args:
        project_id: UUID of the project to delete.

    Returns:
        Dictionary with ok=True on success.
```

#### Task Tools

```python
@mcp.tool()
def tracker_create_task(
    title: str,
    description: str = "",
    project_id: str = "",
    release_id: str = "",
    workstream_id: str = "",
    status: str = "open",
) -> dict:
    """Create a new tracker task.

    Args:
        title: Short, descriptive task title.
        description: Optional longer description. Markdown supported.
        project_id: UUID of the project this task belongs to. Strongly
            recommended — tasks without a project are harder to organize.
        release_id: Optional UUID of the target release.
        workstream_id: Optional FlowTree workstream ID to link this task
            to an active coding workstream.
        status: Task status. Either "open" (default) or "closed".

    Returns:
        Dictionary with ok=True and the created task record.
    """
    _require_scope("write")
    if workstream_id:
        _require_workstream_in_scope(workstream_id)
    _audit("tracker_create_task", project_id=project_id,
           workstream_id=workstream_id)
    payload: dict = {"title": title, "status": status}
    if description:
        payload["description"] = description
    if project_id:
        payload["project_id"] = project_id
    if release_id:
        payload["release_id"] = release_id
    if workstream_id:
        payload["workstream_id"] = workstream_id
    return _tracker_post("/v1/tasks", payload)


@mcp.tool()
def tracker_get_task(task_id: str) -> dict:
    """Get a single tracker task by ID.

    Args:
        task_id: UUID of the task to retrieve.

    Returns:
        Dictionary with ok=True and the task record.
    """
    _require_scope("read")
    _audit("tracker_get_task", task_id=task_id)
    return _tracker_get(f"/v1/tasks/{task_id}")


@mcp.tool()
def tracker_list_tasks(
    project_id: str = "",
    release_id: str = "",
    workstream_id: str = "",
    status: str = "",
    limit: int = 50,
    offset: int = 0,
) -> dict:
    """List tracker tasks with optional filtering.

    Args:
        project_id: Filter to tasks in this project (UUID).
        release_id: Filter to tasks in this release (UUID).
        workstream_id: Filter to tasks linked to this workstream ID.
            Enforces workspace scope — scoped tokens may only query
            workstreams within their scope.
        status: Filter by status: "open" or "closed". Omit for all.
        limit: Maximum number of tasks to return. Defaults to 50,
            maximum 200.
        offset: Pagination offset. Defaults to 0.

    Returns:
        Dictionary with ok=True, a list of tasks, and pagination info
        (total, limit, offset).
    """
    _require_scope("read")
    if workstream_id:
        _require_workstream_in_scope(workstream_id)
    _audit("tracker_list_tasks", project_id=project_id,
           release_id=release_id, workstream_id=workstream_id)
    params = []
    if project_id:
        params.append(f"project_id={project_id}")
    if release_id:
        params.append(f"release_id={release_id}")
    if workstream_id:
        params.append(f"workstream_id={workstream_id}")
    if status:
        params.append(f"status={status}")
    if limit != 50:
        params.append(f"limit={limit}")
    if offset:
        params.append(f"offset={offset}")
    qs = ("?" + "&".join(params)) if params else ""
    return _tracker_get(f"/v1/tasks{qs}")


@mcp.tool()
def tracker_update_task(
    task_id: str,
    title: str = "",
    description: str = "",
    status: str = "",
    project_id: str = "",
    release_id: str = "",
    workstream_id: str = "",
) -> dict:
    """Update an existing tracker task.

    Only fields with non-empty values are updated. Pass an empty string
    to leave a field unchanged.

    To clear an optional field (e.g., remove the release association),
    pass the literal string "null" for that parameter.

    Args:
        task_id: UUID of the task to update.
        title: New title. Omit to leave unchanged.
        description: New description. Omit to leave unchanged.
        status: New status: "open" or "closed". Omit to leave unchanged.
        project_id: New project UUID. Omit to leave unchanged. Pass
            "null" to clear.
        release_id: New release UUID. Omit to leave unchanged. Pass
            "null" to clear.
        workstream_id: New workstream ID. Omit to leave unchanged. Pass
            "null" to clear.

    Returns:
        Dictionary with ok=True and the updated task record.
    """
    _require_scope("write")
    # Scope check: fetch current task, enforce scope on its workstream
    current = _tracker_get(f"/v1/tasks/{task_id}")
    if not current.get("ok"):
        return current
    current_ws = (current.get("task") or {}).get("workstream_id", "")
    if current_ws:
        _require_workstream_in_scope(current_ws)
    if workstream_id and workstream_id != "null":
        _require_workstream_in_scope(workstream_id)
    _audit("tracker_update_task", task_id=task_id)
    payload: dict = {}
    if title:
        payload["title"] = title
    if description:
        payload["description"] = description
    if status:
        payload["status"] = status
    # Handle "null" sentinel to clear optional FK fields
    for field, val in [("project_id", project_id),
                       ("release_id", release_id),
                       ("workstream_id", workstream_id)]:
        if val == "null":
            payload[field] = None
        elif val:
            payload[field] = val
    return _tracker_put(f"/v1/tasks/{task_id}", payload)


@mcp.tool()
def tracker_delete_task(task_id: str) -> dict:
    """Delete a tracker task permanently.

    Args:
        task_id: UUID of the task to delete.

    Returns:
        Dictionary with ok=True on success.
    """
    _require_scope("write")
    # Scope check on the task's workstream before deleting
    current = _tracker_get(f"/v1/tasks/{task_id}")
    if not current.get("ok"):
        return current
    ws = (current.get("task") or {}).get("workstream_id", "")
    if ws:
        _require_workstream_in_scope(ws)
    _audit("tracker_delete_task", task_id=task_id)
    return _tracker_delete(f"/v1/tasks/{task_id}")


@mcp.tool()
def tracker_search_tasks(
    query: str,
    project_id: str = "",
    status: str = "",
    limit: int = 20,
    offset: int = 0,
) -> dict:
    """Full-text search over tracker task titles and descriptions.

    Uses SQLite FTS5 for efficient full-text search. Supports
    phrase queries ("exact phrase"), NOT, AND, OR operators.

    Args:
        query: Search string. Supports FTS5 query syntax.
        project_id: Optional UUID to restrict search to one project.
        status: Optional status filter: "open" or "closed".
        limit: Maximum results to return. Defaults to 20, max 100.
        offset: Pagination offset. Defaults to 0.

    Returns:
        Dictionary with ok=True, a list of matching tasks, and
        pagination info.
    """
    _require_scope("read")
    _audit("tracker_search_tasks", query=query, project_id=project_id)
    params = [f"q={query}"]
    if project_id:
        params.append(f"project_id={project_id}")
    if status:
        params.append(f"status={status}")
    if limit != 20:
        params.append(f"limit={limit}")
    if offset:
        params.append(f"offset={offset}")
    return _tracker_get("/v1/search/tasks?" + "&".join(params))
```

## 5. Jira Migration

### 5.1 Jira CSV Export Format

The standard Jira CSV export (via **Issues → Export → Export Excel CSV (all fields)**
or the equivalent) includes these columns. We only use the ones marked **keep**:

| Jira Column         | Disposition | Maps To             |
|---------------------|-------------|---------------------|
| Issue Key           | keep        | task.id (suffix only, for dedup) |
| Summary             | keep        | task.title          |
| Description         | keep        | task.description    |
| Project Name        | keep        | project.name        |
| Fix Version/s       | keep        | release.name        |
| Status              | keep        | task.status (see mapping below) |
| Created             | keep        | task.created_at     |
| Updated             | keep        | task.updated_at     |
| Assignee            | **drop**    | —                   |
| Reporter            | **drop**    | —                   |
| Priority            | **drop**    | —                   |
| Labels              | **drop**    | —                   |
| Components          | **drop**    | —                   |
| Sprint              | **drop**    | —                   |
| Story Points        | **drop**    | —                   |
| Comments            | **drop**    | —                   |
| Attachment          | **drop**    | —                   |
| Issue Type          | **drop**    | —                   |
| Epic Link           | **drop**    | —                   |
| Parent              | **drop**    | —                   |

**Status mapping** (Jira → tracker):
```python
STATUS_MAP = {
    "Open": "open",
    "To Do": "open",
    "In Progress": "open",
    "In Review": "open",
    "Blocked": "open",
    "Reopened": "open",
    "Done": "closed",
    "Closed": "closed",
    "Resolved": "closed",
    "Won't Do": "closed",
    "Cancelled": "closed",
}
```

**Description handling**: Jira wiki markup is not converted to Markdown. The raw text is
imported as-is. Visual fidelity of historical descriptions is a non-goal; the text is
preserved for searchability.

### 5.2 Migration Script: `tools/tracker/migrate_jira.py`

```
Usage:
  python migrate_jira.py \
    --csv jira_export.csv \
    --tracker-url http://localhost:8030 \
    --tracker-token <token> \
    [--dry-run]
    [--workstream-map workstreams.json]
```

**Algorithm:**

1. **Parse CSV** with `csv.DictReader`. Detect encoding (Jira exports vary; try UTF-8,
   fall back to `latin-1`).

2. **Build projects**: Collect unique `Project Name` values. POST each to
   `/v1/projects` and build a `name → id` map. Existing projects (same name) are
   looked up rather than duplicated.

3. **Build releases**: Collect unique `Fix Version/s` values. A single task may have
   multiple Fix Versions (Jira stores them comma-separated in the CSV); create a release
   for each unique name. POST each to `/v1/releases` with the associated project UUID.

4. **Workstream mapping**: Load `--workstream-map` JSON if provided:
   ```json
   {"PROJ-prefix": "ws-abc123", "summary-keyword": "ws-def456"}
   ```
   For unmatched rows, the script prints a warning and imports the task with
   `workstream_id=null`. A `--interactive` flag can be added later to prompt for
   manual mapping.

5. **Build tasks**: For each CSV row, construct the task payload and accumulate into
   batches of 100.

6. **Bulk import**: POST each batch to `/v1/import`. The import endpoint upserts by
   ID, so re-running the script is safe.

7. **Summary**: Print counts of created/updated/skipped per entity type.

**Task ID strategy**: Generate a deterministic UUID from the Jira Issue Key using
`uuid.uuid5(NAMESPACE_DNS, issue_key)`. This makes re-runs idempotent without querying
the tracker first.

### 5.3 Intentionally Dropped Fields

The following Jira data is permanently dropped at import time. This is by design — we are
simplifying, not archiving:

- **Assignee / Reporter**: FlowTree agents don't have assignees.
- **Comments**: Historical comment threads are not imported.
- **Attachments**: Files attached to Jira issues are not migrated.
- **Sprint / Story Points / Epic hierarchy**: Not part of our model.
- **Labels / Components**: Not part of our model.
- **Priority**: Deferred (see Open Questions §6.5).
- **Issue history / audit log**: Not preserved.
- **Watchers / Votes / Issue links**: Not preserved.

## 6. Open Questions / Decisions Needed

These must be decided before implementation begins.

### 6.1 Task Status: Binary or Multi-State?

The current schema defines `status` as `'open' | 'closed'`. Options:
- **A) Keep binary** — simplest, matches the stated goal of "just enough structure."
  Richer state can be added later.
- **B) Add `in-progress`** — useful for distinguishing "someone is working on this"
  from "nobody has started." Low cost to add now; harder to add after migration.
- **C) Configurable per-project states** — powerful but complex; contradicts the
  minimal design goal.

**Recommendation**: Start with binary (A). Jira status diversity is itself a form of
  process overhead we're trying to escape.

### 6.2 Is `project_id` Required on Tasks?

Options:
- **A) Required at the API level** — simpler queries, cleaner data, breaks Jira import
  for tasks with no project (rare but possible).
- **B) Required at the MCP layer, optional at the API layer** — ar-manager warns if
  omitted but the REST API accepts it. Gives migration flexibility.
- **C) Fully optional** — allows project-less tasks permanently.

**Recommendation**: Option B. Enforce at the MCP layer, accept at the API layer, to
  avoid blocking migration edge cases.

### 6.3 API Authentication Mechanism

Options:
- **A) Shared bearer token** (same as `ar-memory`) — single secret in docker-compose,
  set in `AR_TRACKER_AUTH_TOKEN`. Simple; no token management overhead.
- **B) Same HMAC temporary-token scheme as ar-manager** — agent jobs get scoped tokens
  automatically. Adds token-verification logic to the tracker service.
- **C) No authentication** (network-level isolation only) — acceptable if the service
  is never exposed beyond the Docker network.

**Recommendation**: Option A initially. The memory service uses this and it works. Scope
  enforcement happens in ar-manager, not the tracker API itself.

### 6.4 Database: SQLite vs Postgres

SQLite is recommended (see §1.2). Revisit if:
- Concurrent writes from multiple callers exceed SQLite's WAL-mode throughput (~100
  writes/sec), or
- We add multi-host deployment (SQLite files don't distribute).

For the foreseeable workload (human + agents using MCP tools), SQLite is sufficient.

### 6.5 Task Priority / Ordering

Not included in this design. If added:
- A numeric `priority` field (`1`–`5`) is simplest.
- An orderable linked-list (Jira-style manual drag-to-reorder) is significantly more
  complex.

**Recommendation**: Punt entirely for now. Decide when someone actually needs it.

### 6.6 Task Dependencies / Blocking

Not included. Jira "Linked Issues" are dropped at import. If needed later, a separate
`task_links` table with `source_id`, `target_id`, `link_type` is the natural extension.

## 7. Implementation Phases

### Phase 1: Core API + Docker + MCP Integration

Goal: a working tracker service with basic CRUD, deployed alongside `ar-memory`.

Deliverables:
- `tools/tracker/` — Starlette service (server.py, api.py, store.py, migrate.py)
- `tools/tracker/Dockerfile`
- `ar-tracker` service entry in `flowtree/controller/docker-compose.yml`
- 14 MCP tools added to `tools/mcp/manager/server.py`
- Both tool names and parameter assertions added to `McpToolDiscoveryTest.java` and
  `test_server.py`
- Manual smoke test: create a project, release, and task via MCP

**Not in Phase 1**: Full-text search (deferred to Phase 3 — FTS5 setup is part of the
schema but the search endpoint can return an informative error initially), Jira migration.

### Phase 2: Jira Migration Script

Goal: one-shot import of existing Jira data with zero manual entry.

Deliverables:
- `tools/tracker/migrate_jira.py`
- Documentation of expected CSV format and column mapping
- Dry-run mode tested against a real Jira export
- Idempotency verified (run twice, no duplicates)

### Phase 3: Full-Text Search + Richer Filtering

Goal: make the tracker useful for finding things, not just storing them.

Deliverables:
- FTS5 index and triggers (included in Phase 1 schema, activated here)
- `tracker_search_tasks` MCP tool wired to `/v1/search/tasks`
- `sort` and `order` parameters on list endpoints
- `updated_since` filter for polling patterns (e.g., "show me tasks updated in the
  last 7 days")

### Future / Nice-to-Haves

These are explicitly deferred and should not influence Phase 1–3 scope:
- Task priority / ordering
- Task dependencies
- Markdown rendering preview
- Webhook notifications when tasks change
- Web UI (the MCP interface is the intended UI)
- Slack integration (create tasks from Slack messages)
