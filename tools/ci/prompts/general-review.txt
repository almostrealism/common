### 2. Design and Architecture

- Do the changes follow CLAUDE.md architectural principles (ModelOptimizer owns
  training loops, PackedCollection is not a Java array, StateDictionary for
  weights, etc.)?
- Is the level of abstraction appropriate?
- Is there code duplication that should be extracted into a shared method?
- **Method placement — OOP fundamentals**: Every method must live on the class
  it naturally belongs to. This is the most basic principle of object-oriented
  design. Ask yourself for every new method:
  - Does this method use instance state of its containing class, or could it be
    static? If it could be static, it probably belongs somewhere else — or at
    minimum must be declared static to make its independence explicit.
  - Is this method a general-purpose traversal or query that operates on a base
    type (e.g., `Expression`, `Scope`)? If so, it MUST be defined on that base
    type, not on a subclass. Defining `collectAllReferencedNames(Expression)`
    as a private method of `Repeated` instead of as an instance method on
    `Expression` prevents all other code from reusing it and violates the
    single-responsibility principle.
  - Would this method be useful to other callers? Utility methods for collecting
    declarations from a scope tree, extracting names from assignment
    destinations, or traversing expression trees for references are inherently
    general-purpose. They belong on `Scope` or `Expression`, not buried as
    private helpers in a specific subclass.
  - **Flag as a design violation** if you find methods defined on a subclass
    that: (a) do not reference any instance fields of that subclass, AND
    (b) operate on parameters of a base type. These must be moved to the base
    type or made static utilities on the appropriate class. Fix them — do not
    just flag them.
- **Backwards compatibility vs. clarity**: Ask yourself — are these changes
  preserving a probably-useless kind of backwards compatibility at the expense
  of creating a lot more effort to understand the intended behavior? Prefer a
  "do things once" approach: one method signature, one code path, one way to
  accomplish each task. Duplicate method overloads or parallel "fallback" paths
  that silently degrade to the old behavior are a red flag. If the old behavior
  is genuinely no longer the intended path, remove it or make it throw rather
  than keeping it as a silent fallback. Maintaining two paths doubles the surface
  area for bugs and makes it nearly impossible for the next reader to know which
  path is actually expected to execute in production.

### 3. Test Coverage and Quality (CRITICAL — EQUAL PRIORITY TO TEST INTEGRITY)

**Every modified line of production code MUST be covered by tests.** This is not
optional. If a branch adds or changes production code without corresponding test
coverage, that is a defect you MUST fix before anything else.

Specifically:
- For every changed method: there must be a test that exercises the changed code
  path. If no such test exists, WRITE ONE.
- For every new method or class: there must be tests that verify its behavior.
  If no such tests exist, WRITE THEM.
- For every bug fix: there must be a regression test that fails without the fix
  and passes with it. If no such test exists, WRITE ONE.
- For every new branch or conditional: there must be tests for both the true and
  false paths. If coverage is missing, ADD IT.

**Test quality requirements — every test you write or evaluate must meet ALL of
these criteria:**
- Tests must verify BEHAVIOR, not just that code runs without throwing. A test
  that calls a method and checks no assertions is worthless.
- Tests must include adversarial and boundary cases: null inputs, empty
  collections, maximum/minimum values, off-by-one boundaries, concurrent access
  where applicable, malformed input, and type edge cases.
- Tests must assert on specific expected values, not just "result is not null."
- Tests must be independent and deterministic — no reliance on execution order or
  external state.
- Tests must follow project conventions (extend TestSuiteBase, use @TestDepth
  for expensive tests, include timeout parameters).

**If the branch has production code changes but insufficient test coverage, this
is a blocking issue.** Do not proceed to other review categories until test
coverage is complete. Write the missing tests yourself — you are a coding agent,
not a reviewer.
