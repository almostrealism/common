## What to look for and fix

### 1. Test Integrity (HIGHEST PRIORITY)

Run `git diff ${BASE_BRANCH}...HEAD -- '**/*Test*.java' '**/test/**/*.java'` to
see all test file changes. For each modified test that existed on ${BASE_BRANCH}
(check with `git log --oneline ${BASE_BRANCH} -- <file>`), verify that no
assertions were weakened, no tests were disabled, and no coverage was reduced.
If you find violations: **revert the test modifications and fix the production
code instead** so the original test passes.

### 2. Design and Architecture

- Do the changes follow CLAUDE.md architectural principles (ModelOptimizer owns
  training loops, PackedCollection is not a Java array, StateDictionary for
  weights, etc.)?
- Is the level of abstraction appropriate?
- Is there code duplication that should be extracted into a shared method?
- **Backwards compatibility vs. clarity**: Ask yourself — are these changes
  preserving a probably-useless kind of backwards compatibility at the expense
  of creating a lot more effort to understand the intended behavior? Prefer a
  "do things once" approach: one method signature, one code path, one way to
  accomplish each task. Duplicate method overloads or parallel "fallback" paths
  that silently degrade to the old behavior are a red flag. If the old behavior
  is genuinely no longer the intended path, remove it or make it throw rather
  than keeping it as a silent fallback. Maintaining two paths doubles the surface
  area for bugs and makes it nearly impossible for the next reader to know which
  path is actually expected to execute in production.

### 3. Test Quality

- Do the tests actually prove the new functionality works, or are they
  superficial?
- Are edge cases covered?
- Do the tests follow project conventions (extend TestSuiteBase, use @TestDepth
  for expensive tests, include timeout parameters)?

### 4. Correctness

- Are there logical defects in the implementation?
- Are there fragile assumptions, implicit ordering dependencies, or missing
  validation at system boundaries?
- Is error handling and resource management correct?

### 5. Documentation

- Do all new non-private classes and methods have Javadoc?
- Is the documentation accurate and useful (not just boilerplate)?

### 6. Consistency with Existing Patterns

- Is new code in the correct module?
- Does it duplicate existing functionality? (Search the codebase before
  concluding.)
- Are naming conventions and structural patterns consistent with surrounding
  code?

## How to proceed

1. Identify every issue across the categories above.
2. **Fix each issue directly** by editing the source files.
3. Verify the build still succeeds: `mvn install -DskipTests`
4. Send a Slack summary of what you changed and why. Organize by category.
   For each fix, include the file path and a one-line explanation.

If the changes look good across all categories and nothing needs fixing,
say so explicitly via Slack and explain why you are confident.
